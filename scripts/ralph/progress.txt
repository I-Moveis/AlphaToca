## Codebase Patterns
- Project uses npm (package-lock.json); run `npx tsc --noEmit` for typecheck and `npm test` (vitest) for tests.
- Source lives under `src/` with subdirs `middlewares/`, `services/`, `routes/`, `controllers/`, `utils/`, `config/`; tests in `tests/`.
- Prisma is the ORM (`@prisma/client` 6.4.1). Use `prisma.$transaction` for atomic multi-table writes.
- Validation uses Zod; schemas live in `src/utils/` (e.g., `propertyValidation.ts`).
- `npm install <pkg>` defaults to the absolute latest tag â€” pin to a version range (e.g. `@^1.4.5-lts.2`) when the PRD asks for a specific major line.
- `npm install <pkg>` without `--save-dev` puts the package in `dependencies`, even for `@types/*`. Always pass `--save-dev` for type packages.
- Test suite has 16 pre-existing failures from Auth0 â†’ Firebase migration (tests reference removed `userService.upsertUserFromAuth0`). Not caused by recent stories; verify regressions by running `git stash -u && npm test` vs. current.
- API error response shape across the codebase: `{ status: number, code: string, messages: [{ message: string }] }` (see `src/middlewares/authMiddleware.ts`). Stay consistent when adding new error branches.
- Uploads directory lives at project root. `src/app.ts` uses `path.join(__dirname, '../uploads')`; services under `src/services/` should use `path.join(__dirname, '../../uploads')` so both ts-node (from `src/`) and compiled (from `dist/`) resolve to the same project-root `uploads/`.
- `z.coerce.boolean()` is UNSAFE for multipart request bodies â€” it treats any non-empty string as truthy, so `"false"` becomes `true`. Always use a `z.preprocess` helper that explicitly maps `'true'`â†’true and `'false'`â†’false (pattern already in `src/utils/searchValidation.ts`).
- `z.coerce.number()` is the right way to accept both multipart strings and JSON numbers in the same schema. Compose with `.int()`, `.nonnegative()`, `.positive()` as needed â€” those run AFTER coercion.
- For Prisma `Decimal(10,2)` columns validated via Zod, pattern is `z.coerce.number().<constraint>.transform((v) => new Prisma.Decimal(v))`. Passing `Prisma.Decimal` to `prisma.property.create` works â€” Prisma also accepts number/string but Decimal avoids floating-point drift.
- When a Zod schema uses `.parse(req.body)`, Zod's default object mode is "strip" â€” unknown keys are silently dropped from the output. Adding new OPTIONAL fields to an existing schema never breaks existing callers.
- `prisma.$transaction(async (tx) => ...)` does NOT roll back non-DB side-effects (filesystem writes, external API calls). Wrap the call in an outer try/catch, track what state was created, and compensate in the catch. Inside the callback, use `tx`, never `prisma` â€” mixing them bypasses the transaction.

## 2026-05-06 - US-005
- Updated `propertyService.createProperty` in `src/services/propertyService.ts` to accept an optional `files?: Express.Multer.File[]` as its second argument, and return a new `PropertyWithImages` alias (`Property & { images: PropertyImage[] }`).
- No-files branch: short-circuits to a plain `prisma.property.create({ data, include: { images: true } })`. For JSON-only callers this is functionally unchanged (property is created), with the sole visible difference that the response now has `images: []` â€” which is exactly what US-006 AC requires ("no regression from current JSON-only flow"). No existing caller (propertyController, adminController, whatsappWorker) relied on `images` being absent from the response.
- Files-present branch: wraps everything in `prisma.$transaction(async (tx) => { ... })`:
  1. `tx.property.create({ data })` creates the DB row.
  2. `savePropertyImages(property.id, files)` writes buffers to disk â€” called INSIDE the transaction callback per PRD acceptance criteria, so a DB rollback after this point still triggers filesystem cleanup via the outer catch.
  3. `tx.propertyImage.createMany({ data: saved.map(...) })` inserts the image rows.
  4. `tx.property.findUniqueOrThrow({ where: { id }, include: { images: true } })` returns the full property-with-images payload as the transaction's return value.
- Failure-atomicity: `createdPropertyId` and `savedUrls` are captured on each step that succeeds. On any throw from the transaction, the outer catch calls `cleanupPropertyImages(createdPropertyId, savedUrls)`. A failure INSIDE `savePropertyImages` itself (mid-loop write error, or an INVALID_FILE_TYPE belt-and-suspenders throw) is already self-cleaned by that service â€” `savedUrls` stays `[]` because the assignment only runs after `await` resolves successfully, so the outer cleanup doesn't run twice.
- Cleanup failures are caught and logged via `logger.error`, never thrown â€” we must not mask the original error with a cleanup error.
- Typecheck: `npx tsc --noEmit` passes with zero errors.
- Tests: `npm test` shows 16 failed / 212 passed â€” identical to the pre-existing Auth0-migration baseline. No regressions. (Controller still calls `createProperty(validatedData)` with a single arg; TS allows the new param to be omitted since it's optional.)
- Files changed: `src/services/propertyService.ts` (only).
- **Learnings for future iterations:**
  - Prisma's interactive transaction callback (`prisma.$transaction(async (tx) => {...})`) is the right tool when you need non-DB side-effects (filesystem writes) interleaved with DB writes â€” BUT the side-effects are NOT automatically rolled back. You must do cleanup yourself in an outer catch. The callback receiving `tx` gives you a scoped client; all DB calls inside should use `tx`, never `prisma` (mixing them would bypass the transaction).
  - `findUniqueOrThrow` is preferable to `findUnique` as the final statement in a create-transaction â€” the record must exist (we just created it in the same transaction), so `null` would indicate a serious invariant violation rather than a normal "not found" case. Throwing is the correct signal.
  - When a service accepts an optional trailing parameter (here `files?`), existing callers need NO changes because TypeScript allows omitting optional parameters. This is the safe path for US-006 to wire multer into the controller â€” the signature change is backward-compatible.
  - `Property & { images: PropertyImage[] }` is the pragmatic return type for an `include: { images: true }` Prisma query. `Prisma.PropertyGetPayload<{ include: { images: true } }>` is more formally correct but adds verbosity; both are equivalent at runtime.
  - Ordering of state-capture variables matters: `createdPropertyId = property.id` must come BEFORE `await savePropertyImages(...)` so that a filesystem failure still leaves `createdPropertyId` set â€” but `savedUrls = saved.map(...)` must come AFTER the await resolves, so a throw from savePropertyImages keeps `savedUrls = []` (savePropertyImages handles its own cleanup in that case). This ordering avoids double-cleanup bugs.
  - Cleanup errors in the catch block should be swallowed (or logged at error level) and the ORIGINAL error re-thrown. `cleanupPropertyImages(...).catch(...)` achieves this without a nested try/catch.
