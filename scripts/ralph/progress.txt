## Codebase Patterns
- Seed IDs: random seeds use `crypto.randomUUID()` (not `faker.string.uuid()`); stable demo IDs live in `prisma/demoIds.ts` as canonical UUIDv4 constants — don't replace those constants with `randomUUID()` calls because they're referenced by tests and Firebase UID mappings.
- UUID validation: routes use `z.string().uuid()`; any seed id you introduce must match `/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i`.
- Test runner: `npx vitest run` (vitest config in `vitest.config.ts`). Many pre-existing tests fail due to an unfinished Auth0→Firebase migration (`userService.upsertUserFromAuth0` missing, `tests/auth.test.ts`, `tests/ingestKnowledge.test.ts`) — treat those as baseline noise, not regressions.
- Grep scope for "seed" lookups: include `prisma/` (generators.ts, demoData.ts, demoIds.ts, seed.ts) — seeds are not under `src/` in this repo.
- Typecheck: `npx tsc --noEmit` — must pass before commit.
- `.env` DATABASE_URL points at the live Supabase project — DO NOT run `prisma migrate dev/deploy/reset` locally; hand-author `prisma/migrations/<stamp>_<name>/migration.sql` instead and verify with `npx prisma validate` + `npx prisma generate`. Reviewer runs the actual `migrate reset --force` on a disposable dev DB before landing.
- Swagger is split: `src/config/swagger.ts` owns component schemas (e.g., `Property`); per-route JSDoc blocks in `src/routes/*.ts` own request/response shapes — when an enum changes, both files need an edit.
- Prisma enum value rename: use `ALTER TYPE "X" RENAME VALUE 'OLD' TO 'NEW';` — it rewrites the catalog label in place, so existing rows carry over with zero data loss (no `UPDATE` pass required).
- `propertyValidation.ts` uses `z.nativeEnum(PropertyStatus)` — it re-reads the Prisma client enum at import time, so bumping the enum only requires a `prisma generate`, not a Zod edit.
- Integration tests that need a "backing store" without a real DB: mount the real Express app via `supertest` and swap `propertyService` with `vi.mock('../src/services/propertyService', importOriginal)` spreading the actual module plus overriding only the methods you exercise. A `Map<id, record>` in `beforeEach` is enough to simulate create/update/get/search chains — Prisma model semantics are preserved by having the mock merge updates (`{...existing, ...data}`).
- Route-level auth: `app.use('/api', propertyRoutes)` in src/app.ts mounts WITHOUT the `authStack`, so POST/PUT/DELETE /properties are currently unauthenticated at the Express layer. Owner-only checks (US-006/US-007) will need the middleware added OR an inline `localUser`/ownership guard in the controller.
- Zod error shape: the global errorHandler (`src/middlewares/errorHandler.ts`) already converts `ZodError` into `{ status: 400, code: 'VALIDATION_ERROR', messages: [...] }` — route/controllers only need to `schema.parse(req.body)` and pass the ZodError to `next(error)`.
- `ProcessStatus` on `RentalProcess` in this schema is `TRIAGE|VISIT_SCHEDULED|CONTRACT_ANALYSIS|CLOSED` — there is NO `ACTIVE` value. When a PRD/handoff says "RentalProcess with status=ACTIVE", the semantic the codebase actually implements is the `Contract` model (ContractStatus=ACTIVE|TERMINATED|COMPLETED). `contractService.createContract` already couples an ACTIVE Contract insert with `Property.status='RENTED'` in one transaction — so Contract is the source of truth for "who is the current tenant".
- To add a relation-derived field (e.g., currentTenant) to a property read path without N+1: use `include: { contracts: { where: { status: 'ACTIVE' }, select: { tenant: { select: { id, name } } }, take: 1 } }` in Prisma findUnique/findMany (single JOIN), and for any raw `$queryRaw` path in the same service, emulate the include with ONE batched `prisma.contract.findMany({ where: { status: 'ACTIVE', propertyId: { in: ids } } })` followed by a Map join in memory — still constant queries per page.
- Mocking Prisma for service-level tests: `vi.mock('../src/config/db', () => ({ default: { model: { method: vi.fn() }, $queryRaw: vi.fn() } }))` at the TOP of the test file (before any import that pulls the service in), then cast with `(prisma.model.method as any).mockResolvedValue(...)` per test. See `tests/propertyCurrentTenant.test.ts` for the full template — works cleanly with `src/config/db`'s `export default prisma` shape.
- Mocking `prisma.$transaction(async (tx) => ...)`: the factory returns `{ default: { ..., $transaction: vi.fn(async (fn) => fn({ model: prismaMocks })) } }`. Default impl proxies the `tx` object to the same top-level mocks so callees can use either `tx.model.method` or `prisma.model.method` interchangeably in tests. To simulate a ROLLBACK on mid-tx failure, per-test override with `(prisma.$transaction as any).mockImplementationOnce(async (fn) => { try { return await fn(txProxy); } catch(err) { committed = false; throw err; } })` and assert the caller sees the throw AND the downstream writes never ran. See `tests/rentalLifecycle.test.ts` for the full template.
- Lifecycle coupling semantics in this codebase (US-005): Property.status transitions are coupled to Contract.status, NOT RentalProcess.status. The `ProcessStatus` enum has no ACTIVE value — the source of truth for "tenant is currently renting" is `Contract.status=ACTIVE`. Rules enforced atomically inside Prisma $transactions: (a) `contractService.createContract` — one ACTIVE contract max per property (409 RENTAL_PROCESS_ALREADY_ACTIVE) + Property→RENTED; (b) `contractService.updateContractStatus` — ACTIVE→terminal flips Property→AVAILABLE, terminal→ACTIVE re-checks the guard and flips Property→RENTED; (c) `rentalProcessService.updateStatus` — only releases Property when CLOSED AND property was in NEGOTIATING (i.e., no contract was ever signed); RENTED properties are left alone since contract termination is the authoritative release path.
- Content-Type-conditional multer: to let a single route accept BOTH application/json AND multipart/form-data, wrap `multer.array(...)` in a small handler that only invokes multer when `req.headers['content-type']` starts with `multipart/form-data` (case-insensitive), otherwise `next()`. Multer consumes the request stream, so running it on a JSON body would shadow `express.json()` — never apply it unconditionally. `src/middlewares/uploadMiddleware.ts#conditionalPropertyPhotoUploadHandler` is the reference for this pattern; reuse it for US-007 (photosToRemove) and US-016 (signed PDF) rather than writing a new one.
- Auth gating on the property route group: `app.use('/api', propertyRoutes)` deliberately mounts WITHOUT the global authStack — many GET paths (search, /:id, list) are public. To gate a SPECIFIC route on that router, declare `const authStack = [checkJwt, authSyncMiddleware]` INSIDE `src/routes/propertyRoutes.ts` and spread it per-route (e.g., `router.put('/properties/:id', ...authStack, uploadHandler, controller.update)`). Ownership is enforced INSIDE the controller via `req.localUser.id === property.landlordId` — this avoids the 404-vs-403 leak (404 first if the property doesn't exist, then 403 if it does but you're not the owner).
- Cover-preservation rule on image-adding updates: when new photos are uploaded alongside a PUT that keeps existing images, NEVER promote a new photo to cover unless the property has zero covers. Pattern: `const hasExistingCover = existing.images.some(img => img.isCover)` read before the tx, then in the `propertyImage.createMany` step compute `isCover: !hasExistingCover && idx === 0`. Don't reuse `savePropertyImages`' default `isCover: i === 0` because it would silently replace the existing cover.
- Multer text fields coexist with file arrays: `multer.array('photos', 20)` accepts the FILE field `photos` AND collects any repeated non-file text fields into `req.body` (single occurrence → string, 2+ → string[]). No need for `.fields()` or a second handler — the existing `propertyPhotoUploadHandler` already routes `photosToRemove` text fields into `req.body.photosToRemove`. Normalize the string-vs-string[] shape in Zod via `z.preprocess(v => typeof v === 'string' ? [v] : v, z.array(...))` so downstream code sees a uniform array.
- `PropertyError` lives in `src/services/propertyService.ts` and mirrors `ContractError`/`VisitError`/`ProposalError` — `httpStatus, code, message`. The controller pattern is `if (err instanceof PropertyError) res.status(err.httpStatus).json({ status, code, messages: [{ message }] })` right inside the try/catch, BEFORE `next(err)`. Reuse this class for any future 4xx business rule in property service (e.g., "cannot delete property with active contract"). Don't add a new error class for each story.
- Relative-path URLs in schema: `PropertyImage.url` is stored as `/uploads/<propertyId>/<file>.jpg` — NOT an absolute `https://...` URL. Zod validation for any field carrying this value must use `z.string().min(1)` (or a path regex), NOT `z.string().url()`. Check the actual stored shape via `prisma/schema.prisma` + the producer service (`propertyImageStorageService.savePropertyImages`) before picking a Zod primitive.
- Post-commit storage cleanup is fire-and-forget: after a Prisma `$transaction` that modifies rows linked to files on disk, run `await cleanupPropertyImages(id, urls).catch(err => logger.warn({err, propertyId: id}, '...'))` OUTSIDE the tx so a storage failure never rolls back committed DB state. The cleanup function already tolerates ENOENT and non-empty-dir `rmdir`, so it's safe to pass a mix of existing + already-gone URLs. Contrast with the pre-commit path inside `savePropertyImages`'s outer try/catch which DOES throw on the way up to trigger the tx rollback cleanup.
- Mocking `propertyImageStorageService` for service-level tests: `vi.mock('../src/services/propertyImageStorageService', () => ({ savePropertyImages: vi.fn(), cleanupPropertyImages: vi.fn() }))` at the top of the test file, combined with the standard `vi.mock('../src/config/db', ...)` $transaction pattern, exercises the full `updateProperty` tx path end-to-end without disk I/O. The only caveat: `Buffer` + a minimal fake `Express.Multer.File` object (fieldname/mimetype/buffer) is enough to flow through the service's type expectations since the real multer mime inspection happens upstream at the middleware layer.

## [2026-05-07] - US-005 - Auto-transition Property.status on RentalProcess lifecycle
- Semantic mapping decided up-front (mirrors the US-004 learning): this codebase does not carry an `ACTIVE` state on `RentalProcess` (ProcessStatus=TRIAGE|VISIT_SCHEDULED|CONTRACT_ANALYSIS|CLOSED). The "tenant is currently renting" relation is modeled by `Contract.status=ACTIVE`, and `contractService.createContract` is already the one place that flips `Property.status='RENTED'` atomically. So the auto-transition logic for US-005 spans two services:
  1. `contractService.createContract` — acts as "activation". Added the 409 RENTAL_PROCESS_ALREADY_ACTIVE guard as the first operation inside the existing `prisma.$transaction(async (tx) => { ... })` callback: `tx.contract.findFirst({ where: { propertyId, status: 'ACTIVE' } })` — if a row is found, throw `ContractError(409, 'RENTAL_PROCESS_ALREADY_ACTIVE', ...)` before any write. Prisma's tx semantics guarantee the throw rolls back any prior write in the callback.
  2. `contractService.updateContractStatus(id, newStatus)` — new, acts as "termination + re-activation". Runs inside `prisma.$transaction`: 404 CONTRACT_NOT_FOUND if the contract is missing; same-state write is a no-op; on `ACTIVE → TERMINATED/COMPLETED` the same tx sets `Property.status='AVAILABLE'`; on `terminal → ACTIVE` it re-runs the 409 guard (excluding the current row) and flips `Property.status='RENTED'`.
  3. `rentalProcessService.updateStatus` — wraps the existing `prisma.rentalProcess.update` in `prisma.$transaction(async (tx) => { ... })`. When `newStatus==='CLOSED'` AND the cached property status is `NEGOTIATING` (no contract signed yet), the same tx resets `Property.status='AVAILABLE'`. `RENTED` is intentionally left alone — contract termination is the authoritative release path. `VISIT_SCHEDULED`, `CONTRACT_ANALYSIS`, etc. never trigger property writes.
- New API surface: `PATCH /api/contracts/:id/status` route (mounted with `authStack` via the existing `app.use('/api', authStack, contractRoutes)` block) → `contractController.updateStatus` → service. Uses the already-existing `updateContractStatusSchema` from `src/utils/contractValidation.ts` (Zod wraps `ContractStatus` via `z.nativeEnum`).
- Swagger: `Property.status` gets a `description` listing the three endpoints that can mutate it as a side-effect (POST /contracts, PATCH /contracts/:id/status, PATCH /rental-process/:id/status). All three route JSDoc blocks were already in place — only the central component description was expanded. Route-level JSDoc for the new PATCH /contracts/:id/status was added to `src/routes/contractRoutes.ts`.
