## Codebase Patterns
(Carry over from archive/2026-05-07-landlord-backend-closeout/progress.txt — add new ones as discovered during this run.)

- **Auth middleware mock**: tests that exercise 401/403 use the `importOriginal` pattern to partially mock `authMiddleware` (keeps `requireRole` real, swaps `checkJwt`/`authSyncMiddleware` with test doubles that inject a fake `req.localUser`). Apply when writing new auth-guarded tests.
- **Hand-author migrations**: NEVER run `prisma migrate dev`. Write `prisma/migrations/<stamp>_<slug>/migration.sql` by hand, then verify byte-for-byte with `npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script`.
- **Decimal deserialization**: Prisma `@db.Decimal(10,2)` columns come back as `Prisma.Decimal` — convert to plain number with `Number(row.field)` before putting them in HTTP response bodies.
- **PropertyImage.url format is inconsistent by origin**: `propertyImageStorageService.savePropertyImages` writes RELATIVE URLs (`/uploads/<propertyId>/<uuid>.png`), but `prisma/demoData.ts` seeds ABSOLUTE URLs (`http://10.0.2.2:3000/uploads/<propertyId>/NNNN.png`). Frontend `absoluteImageUrl()` + `_normalizeLocalHost()` handle both. Keep it in mind when adding new image endpoints.
- **CORS + static ordering in src/app.ts**: `app.use(cors())` is at line 40, BEFORE `app.use('/uploads', express.static(...))` at line 50. `/uploads/*` responses include `Access-Control-Allow-Origin: *`. Do NOT move static before cors.
- **Pre-existing test failures (baseline)**: On `ralph/backend-gaps-finais`, `npx vitest run` reports **23 failing / 618 passing / 65 files** with a CLEAN working tree. The 6 failing files are: `tests/auth.test.ts`, `tests/ingestKnowledge.test.ts`, `tests/ragChainService.test.ts`, `tests/ragRetrieverService.test.ts`, `tests/userSync.test.ts`, `tests/visitValidation.test.ts`. These pre-date this rollout (RAG/ingest/auth plumbing drift) and are unrelated to `/uploads`, CORS, or the P4 user stories. Future stories should treat this count as the baseline — a story is non-regressive if it leaves 23 or fewer failures in these same 6 files.
- **Support ticket shape split (admin vs user view)**: `supportTicketService` now exposes two list shapes — `SupportTicketAdminView` (with `user`, `assignedTo`, `resolution`, `updatedAt`) for `GET /admin/support/tickets`, and `SupportTicketUserView` (just `id`, `code`, `title`, `description`, `createdAt`, `status`) for `GET /support/tickets`. Keep them separate: the user view intentionally hides triage metadata from the ticket's own opener. When adding new fields, decide which view(s) need them and update the relevant `select:` or `toAdminView` builder — don't merge the two.
- **Support routes mount**: `src/routes/supportRoutes.ts` is mounted once via `app.use('/api', authStack, supportRoutes)` in src/app.ts. Every route defined in this file (including `/support/tickets`, `/admin/support/tickets`) is therefore behind `authStack` (checkJwt + authSyncMiddleware). Admin-only routes layer `requireRole('ADMIN')` inside the router itself. When adding a new support endpoint, do NOT re-mount the router in app.ts — add the route in supportRoutes.ts and (if it needs admin) gate with `adminOnly`.
- **Check whether a P4 endpoint was already built in an earlier LL-rollout before writing code**. LL-011→LL-015 delivered the full conversations/messages API (GET /conversations, GET /conversations/resolve, GET /conversations/:id/messages, POST /conversations/:id/messages, POST /conversations/:id/read). US-004→US-006 in this PRD overlap word-for-word with those endpoints. Always grep `src/routes/` + `src/controllers/` for the path before starting the story — if it exists, run `git log --oneline -- <file>` to see which archived rollout produced it. If shape + guards + tests already match the AC, close the story as a no-op in the PRD `notes` and skip the code churn. Do NOT rewrite equivalent logic.
- **Conversation schema uses landlordId/tenantId columns, not a `participants[]` array**. When an AC asks to "exclude req.localUser.id from participants", translate that to `WHERE landlordId = userId OR tenantId = userId` + `counterpart = (row.landlordId === userId ? row.tenant : row.landlord)`. The schema is intentionally denormalized this way (single UNIQUE index on `(propertyId, landlordId, tenantId)` anchors the "one 1:1 thread per property per pair" invariant). Do NOT add a participants array — it would duplicate state and break the upsert contract in `conversationService.resolve`.
- **Existence-hiding 404 is the project-wide convention for conversation endpoints, not 403**. LL-012, LL-013, and LL-015 (`GET/POST /conversations/:id/messages`, `POST /conversations/:id/read`) all return 404 for both "conversation doesn't exist" AND "caller is not a participant" — never 403. This is a deliberate security posture: a 403 vs 404 oracle lets an authenticated attacker enumerate valid conversation UUIDs by probing. When a P4 AC says "403 for non-participant" for a conversation endpoint, prefer the existence-hiding 404 that's already locked by tests and JSDoc — don't downgrade the guard to match AC wording. Update the PRD notes to document the deviation.
- **Chat pagination returns ASC-within-a-page even though DB orderBy is DESC**. LL-012's `conversationService.listMessages` queries `orderBy: { createdAt: 'desc' }` + `take: limit` (efficient for "latest N" / "N before cursor"), then does `rows.reverse()` in-memory to hand the client ASC within the page. The cursor pagination DIRECTION (older pages via `before`) is still DESC-of-pages. When an AC says "ordered createdAt DESC", translate to "paginated DESC-of-pages but items within a page are ASC" — that's the contract the frontend chat UI expects (oldest-at-top, newest-at-bottom). Tests in `tests/conversationMessagesService.test.ts` lock this in.
- **Cursor-forging guard in listMessages**: when `before` is provided, `findUnique({ where: { id: before } })` is called and the result's `conversationId` is compared against the path's `:id`. If they differ (or the cursor doesn't exist), the service returns `{ messages: [], markedReadIds: [] }` — "end of list" semantics. This prevents a caller from passing a message id from ANOTHER conversation as the `before` cursor to probe timestamps. When adding new cursor-based endpoints, replicate this guard.
- **`prisma migrate diff` without a shadow DB**: this repo's `.env` DATABASE_URL points at Supabase remote (no local shadow available). The Codebase Patterns line "verify byte-for-byte with `--from-empty --to-schema-datamodel`" gives the full schema DDL, which is noisy when you only want the new-change diff. Instead: `git show HEAD:prisma/schema.prisma > /tmp/schema-before.prisma && npx prisma migrate diff --from-schema-datamodel /tmp/schema-before.prisma --to-schema-datamodel prisma/schema.prisma --script`. That emits ONLY the ALTER/CREATE statements for the current uncommitted edits — which is exactly what you want to paste into the hand-authored `migration.sql` before adding any backfill UPDATEs.
- **$transaction mock pattern for service tests**: when a service uses `prisma.$transaction(async (tx) => {...})`, mock prisma so that `$transaction` invokes the callback with a `tx` object that PROXIES the SAME mocks the test asserts against. Example: `const messageCreate = vi.fn(); const $transaction = vi.fn(async (fn) => fn({ conversationMessage: { create: messageCreate } }))`. This lets a single `expect(messageCreate).toHaveBeenCalled(...)` assertion catch calls made inside the tx callback, without the test needing to distinguish tx-vs-non-tx prisma handles. Pattern lives in `tests/propertyPhotosRemove.test.ts` (used for $transaction on property photos) and now `tests/conversationMessageCreateService.test.ts` (createMessage + Conversation.lastMessageAt update).
- **Conversation.lastMessageAt is now the authoritative ordering timestamp** (US-006). Nullable column populated by `conversationService.createMessage` inside a `$transaction` with the INSERT. `conversationService.list` prefers `row.lastMessageAt ?? lastMsg?.createdAt ?? row.createdAt`. When adding new chat/thread endpoints that need "last activity" ordering, read the column directly; do NOT re-introduce an `include: { messages: { take: 1 } }` subquery. Empty threads keep `lastMessageAt = null` — that's intentional (distinct from "has one ancient message"); the list fallback handles the sort correctly via `createdAt`.
- **Injectable `now: Date = new Date()` for any time-branching service** (US-007 `rentalPaymentService.listByTenant`, also used by `getCurrent`/`upsertCurrent`). Real callers never pass the third arg — tests pin it to exercise deterministic "past vs current month" branches. Apply this pattern to US-010 metrics (30-day window) and US-011 analytics (30d/90d/1y windows) — do NOT call `new Date()` inline in a service that gets tested.
- **Month comparison via lexicographic string compare**: `period < currentPeriod` is the correct and cheapest way to check "is this month in the past". `YYYY-MM` strings sort chronologically because of the fixed-width format. No Date parsing needed in hot paths. Same trick applies to `YYYY-MM-DD` — avoid `Date.parse` unless you actually need arithmetic.
- **Synthesis pattern for "one row per month" histories**: when an AC asks for a dense monthly series (payments history, analytics dailies), build the period list FIRST from a source-of-truth (contract window, query window), then LEFT-JOIN-style merge DB rows on top. Missing periods get synthesized with deterministic defaults. The reverse (start from DB rows and fill gaps) is always more code and more bugs. Pattern lives in `rentalPaymentService.listByTenant` (US-007) — copy for US-010/US-011.

### So what IS breaking the placeholder on the frontend?

Out of scope for this US (backend only), but documenting so US-002 doesn't spin its wheels:

1. **`_normalizeLocalHost` is too aggressive for Android emulator**. Seed rows store `http://10.0.2.2:3000/uploads/...`, which IS the correct host alias when the Flutter app runs on an Android emulator. The frontend's `_normalizeLocalHost` (per GAPS_FINAIS.md §1.4) rewrites `10.0.2.2` → `localhost`, which is valid for Flutter web but breaks on the Android emulator (where `localhost` points to the emulator itself, not the host machine). Both work today thanks to the `absoluteImageUrl` helper, but aligning on relative-only (and letting the client prepend its configured backend host) would remove an entire class of host-resolution bugs. Tracked separately — do not do it as part of US-002.

### Learnings for future iterations

- **LL-016 decoupled `documentStatus` from `status`** intentionally. The GAPS_FINAIS.md §2.5 AC description talks about replacing "the heuristic based on property.status" for the 'Meus Inquilinos' chip — so `Contract.documentStatus` is a separate lifecycle from `Contract.status` (ACTIVE/TERMINATED/COMPLETED). An ACTIVE contract can have ANY documentStatus (the tenant is living there but the documents might not be signed yet); a TERMINATED contract typically has APPROVED (they eventually signed before moving out). Don't conflate the two when writing metrics or UI filters downstream (e.g., US-010 proposals count should key off Proposal.status, not Contract.documentStatus).
- **Grep for the field across layers, not just the schema**, when evaluating whether a story is a no-op: `grep -rn documentStatus tests/ src/ prisma/` returned 30+ hits across migration, service, controller, validation, tests, and Swagger — LL-016 was wired end-to-end. If only the schema mentioned it, the story would still have real work (wiring). The 30-hit surface is the tell that a prior rollout delivered it.
- **US-009 (echo title/description in POST ticket response) is pure controller glue** — no schema, no migration, no service changes. Should be the fastest story in this batch. Before starting, `grep -n createTicket src/controllers/supportTicketController.ts` + `cat tests/supportTicketCreate.test.ts | head -80` to see the current response shape, then it's a ~5-line edit + test update.
- **US-010→US-014 are the harder ones** — US-010 (landlord metrics) and US-011 (per-property analytics) need real queries across multiple models with 30d/90d/1y windows (remember the injectable `now` pattern from US-007). US-013 (Notification model) is a fresh migration + new model + persist-on-broadcast side effect. US-015 needs `ALTER TYPE ADD VALUE` migrations which CANNOT run in a transaction block (per the AC's explicit callout — `migration.sql` must not wrap them in BEGIN/COMMIT).
