  - Prisma's interactive transaction callback (`prisma.$transaction(async (tx) => {...})`) is the right tool when you need non-DB side-effects (filesystem writes) interleaved with DB writes â€” BUT the side-effects are NOT automatically rolled back.   - Ordering of branches in `errorHandler` matters: `instanceof` class checks (ZodError, multer.MulterError) should come BEFORE message-string checks (INVALID_FILE_TYPE) and generic `.name` sniffing (SyntaxError). Most-specific first, generic-fallback last. When adding a new branch, pick the spot that preserves this gradient â€” don't just append.
  - `multer.MulterError` is a class exported from the top-level `multer` import, and `err.code` is a union of string literals (`'LIMIT_FILE_SIZE'`, etc.). Prefer a lookup table over a long switch â€” it keeps the handler declarative and easy to extend when multer adds new codes.
  - Always include a fallback inside an instanceof branch (`mapped?.code ?? 'UPLOAD_ERROR'`). A multer minor-version upgrade could introduce a new `err.code` that isn't in the map; without a fallback the branch would crash into `response.messages = [{ message: undefined }]` and confuse clients. Defensive mapping costs one line.
  - When testing error-handler response shapes WITHOUT a network stack, supertest on a bare `express()` app with synthetic `next(err)` routes is the cleanest approach. No need to wire up real multer or real routes to verify the mapping â€” just construct the error class directly: `new multer.MulterError('LIMIT_FILE_SIZE', 'photos')`. That's faster than sending actual >10MB buffers through a real upload.
  - For internationalization: the PRD mandates Portuguese message strings for multer errors (`'Arquivo excede 10MB'`, etc.). If this API ever adds a locale layer, these strings should move to a message catalog keyed by `code`. For now, inline is fine â€” all other API error messages in this codebase are also hardcoded.
