## Codebase Patterns
- Prisma enum names are globally unique in a schema ‚Äî you cannot define two enums with the same name. `PaymentStatus` is already taken by `TenantPayment` (PENDING|PAID|OVERDUE|CANCELLED); the monthly rent status enum lives under `RentalPaymentStatus` (AWAITING|PAID|LATE) on `RentalPayment`. If a PRD asks for an enum name that's already bound, pick a namespaced alternative (`<Model>Status`) and document the deviation ‚Äî don't rename the incumbent.
- Hand-authored migration verification: before committing a hand-authored `prisma/migrations/<stamp>_<name>/migration.sql`, run `npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script` and grep for your table name. The generated DDL for the new table/index/FK should match the hand-authored file byte-for-byte (ENUM CREATE order, VARCHAR(n) vs TEXT, DEFAULT 'X', index/constraint names, CASCADE vs SET NULL on FK). `prisma migrate diff` is read-only and safe to run against any URL ‚Äî including the live Supabase pooler ‚Äî since it operates on the schema file, not the DB.
- Relation onDelete conventions in this schema: child-of-Property tables use `onDelete: Cascade` (property_images, favorites, property_expenses, maintenance_logs, notifications, rental_payments) so a property delete takes the subgraph with it. User-is-actor FKs (e.g., `RentalPayment.updatedBy`) use `onDelete: SET NULL` so deleting a user preserves historical rows with a null actor. Avoid `Restrict` unless the business rule actively forbids deletion (contracts use Restrict because a pending contract should block landlord/property/tenant deletion).
- Seed IDs: random seeds use `crypto.randomUUID()` (not `faker.string.uuid()`); stable demo IDs live in `prisma/demoIds.ts` as canonical UUIDv4 constants ‚Äî don't replace those constants with `randomUUID()` calls because they're referenced by tests and Firebase UID mappings.
- UUID validation: routes use `z.string().uuid()`; any seed id you introduce must match `/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i`.
- Test runner: `npx vitest run` (vitest config in `vitest.config.ts`). Many pre-existing tests fail due to an unfinished Auth0‚ÜíFirebase migration (`userService.upsertUserFromAuth0` missing, `tests/auth.test.ts`, `tests/ingestKnowledge.test.ts`) ‚Äî treat those as baseline noise, not regressions.
- Grep scope for "seed" lookups: include `prisma/` (generators.ts, demoData.ts, demoIds.ts, seed.ts) ‚Äî seeds are not under `src/` in this repo.
- Typecheck: `npx tsc --noEmit` ‚Äî must pass before commit.
- `.env` DATABASE_URL points at the live Supabase project ‚Äî DO NOT run `prisma migrate dev/deploy/reset` locally; hand-author `prisma/migrations/<stamp>_<name>/migration.sql` instead and verify with `npx prisma validate` + `npx prisma generate`. Reviewer runs the actual `migrate reset --force` on a disposable dev DB before landing.
- Swagger is split: `src/config/swagger.ts` owns component schemas (e.g., `Property`); per-route JSDoc blocks in `src/routes/*.ts` own request/response shapes ‚Äî when an enum changes, both files need an edit.
- Prisma enum value rename: use `ALTER TYPE "X" RENAME VALUE 'OLD' TO 'NEW';` ‚Äî it rewrites the catalog label in place, so existing rows carry over with zero data loss (no `UPDATE` pass required).
- `propertyValidation.ts` uses `z.nativeEnum(PropertyStatus)` ‚Äî it re-reads the Prisma client enum at import time, so bumping the enum only requires a `prisma generate`, not a Zod edit. Same convention for `rentalPaymentValidation.ts` ‚Üí `RentalPaymentStatus` (US-010).
- Integration tests that need a "backing store" without a real DB: mount the real Express app via `supertest` and swap `propertyService` with `vi.mock('../src/services/propertyService', importOriginal)` spreading the actual module plus overriding only the methods you exercise. A `Map<id, record>` in `beforeEach` is enough to simulate create/update/get/search chains ‚Äî Prisma model semantics are preserved by having the mock merge updates (`{...existing, ...data}`).
- Route-level auth: `app.use('/api', propertyRoutes)` in src/app.ts mounts WITHOUT the `authStack`, so POST/PUT/DELETE /properties are currently unauthenticated at the Express layer. Owner-only checks (US-006/US-007) will need the middleware added OR an inline `localUser`/ownership guard in the controller.
- Zod error shape: the global errorHandler (`src/middlewares/errorHandler.ts`) already converts `ZodError` into `{ status: 400, code: 'VALIDATION_ERROR', messages: [...] }` ‚Äî route/controllers only need to `schema.parse(req.body)` and pass the ZodError to `next(error)`.
- `ProcessStatus` on `RentalProcess` in this schema is `TRIAGE|VISIT_SCHEDULED|CONTRACT_ANALYSIS|CLOSED` ‚Äî there is NO `ACTIVE` value. When a PRD/handoff says "RentalProcess with status=ACTIVE", the semantic the codebase actually implements is the `Contract` model (ContractStatus=ACTIVE|TERMINATED|COMPLETED). `contractService.createContract` already couples an ACTIVE Contract insert with `Property.status='RENTED'` in one transaction ‚Äî so Contract is the source of truth for "who is the current tenant".
- To add a relation-derived field (e.g., currentTenant) to a property read path without N+1: use `include: { contracts: { where: { status: 'ACTIVE' }, select: { tenant: { select: { id, name } } }, take: 1 } }` in Prisma findUnique/findMany (single JOIN), and for any raw `$queryRaw` path in the same service, emulate the include with ONE batched `prisma.contract.findMany({ where: { status: 'ACTIVE', propertyId: { in: ids } } })` followed by a Map join in memory ‚Äî still constant queries per page.
- Mocking Prisma for service-level tests: `vi.mock('../src/config/db', () => ({ default: { model: { method: vi.fn() }, $queryRaw: vi.fn() } }))` at the TOP of the test file (before any import that pulls the service in), then cast with `(prisma.model.method as any).mockResolvedValue(...)` per test. See `tests/propertyCurrentTenant.test.ts` for the full template ‚Äî works cleanly with `src/config/db`'s `export default prisma` shape.
- Mocking `prisma.$transaction(async (tx) => ...)`: the factory returns `{ default: { ..., $transaction: vi.fn(async (fn) => fn({ model: prismaMocks })) } }`. Default impl proxies the `tx` object to the same top-level mocks so callees can use either `tx.model.method` or `prisma.model.method` interchangeably in tests. To simulate a ROLLBACK on mid-tx failure, per-test override with `(prisma.$transaction as any).mockImplementationOnce(async (fn) => { try { return await fn(txProxy); } catch(err) { committed = false; throw err; } })` and assert the caller sees the throw AND the downstream writes never ran. See `tests/rentalLifecycle.test.ts` for the full template.
- Lifecycle coupling semantics in this codebase (US-005): Property.status transitions are coupled to Contract.status, NOT RentalProcess.status. The `ProcessStatus` enum has no ACTIVE value ‚Äî the source of truth for "tenant is currently renting" is `Contract.status=ACTIVE`. Rules enforced atomically inside Prisma $transactions: (a) `contractService.createContract` ‚Äî one ACTIVE contract max per property (409 RENTAL_PROCESS_ALREADY_ACTIVE) + Property‚ÜíRENTED; (b) `contractService.updateContractStatus` ‚Äî ACTIVE‚Üíterminal flips Property‚ÜíAVAILABLE, terminal‚ÜíACTIVE re-checks the guard and flips Property‚ÜíRENTED; (c) `rentalProcessService.updateStatus` ‚Äî only releases Property when CLOSED AND property was in NEGOTIATING (i.e., no contract was ever signed); RENTED properties are left alone since contract termination is the authoritative release path.
- Content-Type-conditional multer: to let a single route accept BOTH application/json AND multipart/form-data, wrap `multer.array(...)` in a small handler that only invokes multer when `req.headers['content-type']` starts with `multipart/form-data` (case-insensitive), otherwise `next()`. Multer consumes the request stream, so running it on a JSON body would shadow `express.json()` ‚Äî never apply it unconditionally. `src/middlewares/uploadMiddleware.ts#conditionalPropertyPhotoUploadHandler` is the reference for this pattern; reuse it for US-007 (photosToRemove) and US-016 (signed PDF) rather than writing a new one.
- Auth gating on the property route group: `app.use('/api', propertyRoutes)` deliberately mounts WITHOUT the global authStack ‚Äî many GET paths (search, /:id, list) are public. To gate a SPECIFIC route on that router, declare `const authStack = [checkJwt, authSyncMiddleware]` INSIDE `src/routes/propertyRoutes.ts` and spread it per-route (e.g., `router.put('/properties/:id', ...authStack, uploadHandler, controller.update)`). Ownership is enforced INSIDE the controller via `req.localUser.id === property.landlordId` ‚Äî this avoids the 404-vs-403 leak (404 first if the property doesn't exist, then 403 if it does but you're not the owner).
- Cover-preservation rule on image-adding updates: when new photos are uploaded alongside a PUT that keeps existing images, NEVER promote a new photo to cover unless the property has zero covers. Pattern: `const hasExistingCover = existing.images.some(img => img.isCover)` read before the tx, then in the `propertyImage.createMany` step compute `isCover: !hasExistingCover && idx === 0`. Don't reuse `savePropertyImages`' default `isCover: i === 0` because it would silently replace the existing cover.
- Multer text fields coexist with file arrays: `multer.array('photos', 20)` accepts the FILE field `photos` AND collects any repeated non-file text fields into `req.body` (single occurrence ‚Üí string, 2+ ‚Üí string[]). No need for `.fields()` or a second handler ‚Äî the existing `propertyPhotoUploadHandler` already routes `photosToRemove` text fields into `req.body.photosToRemove`. Normalize the string-vs-string[] shape in Zod via `z.preprocess(v => typeof v === 'string' ? [v] : v, z.array(...))` so downstream code sees a uniform array.
- `PropertyError` lives in `src/services/propertyService.ts` and mirrors `ContractError`/`VisitError`/`ProposalError` ‚Äî `httpStatus, code, message`. The controller pattern is `if (err instanceof PropertyError) res.status(err.httpStatus).json({ status, code, messages: [{ message }] })` right inside the try/catch, BEFORE `next(err)`. Reuse this class for any future 4xx business rule in property service (e.g., "cannot delete property with active contract"). Don't add a new error class for each story.
- Relative-path URLs in schema: `PropertyImage.url` is stored as `/uploads/<propertyId>/<file>.jpg` ‚Äî NOT an absolute `https://...` URL. Zod validation for any field carrying this value must use `z.string().min(1)` (or a path regex), NOT `z.string().url()`. Check the actual stored shape via `prisma/schema.prisma` + the producer service (`propertyImageStorageService.savePropertyImages`) before picking a Zod primitive.
- Post-commit storage cleanup is fire-and-forget: after a Prisma `$transaction` that modifies rows linked to files on disk, run `await cleanupPropertyImages(id, urls).catch(err => logger.warn({err, propertyId: id}, '...'))` OUTSIDE the tx so a storage failure never rolls back committed DB state. The cleanup function already tolerates ENOENT and non-empty-dir `rmdir`, so it's safe to pass a mix of existing + already-gone URLs. Contrast with the pre-commit path inside `savePropertyImages`'s outer try/catch which DOES throw on the way up to trigger the tx rollback cleanup.
- Mocking `propertyImageStorageService` for service-level tests: `vi.mock('../src/services/propertyImageStorageService', () => ({ savePropertyImages: vi.fn(), cleanupPropertyImages: vi.fn() }))` at the top of the test file, combined with the standard `vi.mock('../src/config/db', ...)` $transaction pattern, exercises the full `updateProperty` tx path end-to-end without disk I/O. The only caveat: `Buffer` + a minimal fake `Express.Multer.File` object (fieldname/mimetype/buffer) is enough to flow through the service's type expectations since the real multer mime inspection happens upstream at the middleware layer.
- Period-computation convention (US-009+): the server-side `YYYY-MM` of the current month is computed in UTC via `new Date().getUTCFullYear()` + `getUTCMonth()+1` zero-padded. Pattern is centralized as `currentPeriod(now?: Date): string` in `src/services/rentalPaymentService.ts` ‚Äî the US-010 upsert also consumes it so GET/PUT handlers agree byte-for-byte on month boundaries. Never accept a `period` from the request body/query ‚Äî the PRD contract is that the client does not dictate the period (blocks retroactive edits via API).
- Prisma `upsert` on a compound-unique key: the `where` clause reuses the SAME client key used by `findUnique` (the `@@unique([...], name: "...")` value, e.g. `rental_payments_property_period_key`). The `create` payload must include all NOT-NULL columns not defaulted by Prisma (`propertyId`, `period` here ‚Äî `status` is defaulted, `updatedAt` is `@updatedAt`). The `update` payload only carries the mutable fields (`status`, `updatedBy`). `@updatedAt` fires automatically on BOTH create and update ‚Äî do NOT set it from the service or the mock tests would have to fake it.
- "GET returns default without persisting" pattern for compound-unique lookups: when a row may not exist yet but the UI needs a stable default shape (US-009), the service does `prisma.X.findUnique({ where: { <compound_key>: { ... } } })` and branches on `null` ‚Üí returns the default in-memory instead of calling `upsert`/`create`. The write-on-read anti-pattern is easy to fall into when "GET a resource that doesn't exist" feels wrong ‚Äî but the PRD contract here is explicit: AWAITING is the zero-state, not an inserted row. Keep writes in the PUT path only.
- Route protection on `propertyRoutes.ts` (added in US-006): the local `const authStack = [checkJwt, authSyncMiddleware]` is now the canonical way to gate a single route on the publicly-mounted router. Reuse it for US-009/US-010 (`/payments/current`) and US-012 (`/conversations/resolve` if that lives under the same mount) ‚Äî don't reintroduce `app.use('/api', authStack, propertyRoutes)` globally or you'll break the public GET /search + GET /:id paths.

## [2026-05-07] - US-011 - Add Conversation model + migration
- Schema-only story setting up the storage US-012 (GET /api/conversations/resolve) will upsert into. Chose `Conversation` (singular, Pascal) as the model name matching the PRD's Swagger component `Conversation`; `@@map("conversations")` for the table.
- Model fields per the PRD contract: `id String @id @default(uuid())`, `propertyId @map("property_id")`, `landlordId @map("landlord_id")`, `tenantId @map("tenant_id")`, `createdAt @default(now()) @map("created_at")`. No enum, no updatedAt ‚Äî the row is immutable once created (US-012's upsert returns the existing row when the compound key exists; no fields to update).
- Compound unique: `@@unique([propertyId, landlordId, tenantId], name: "conversations_property_landlord_tenant_key")`. The `name:` sets the CLIENT-side compound-where key (`prisma.conversation.findUnique({ where: { conversations_property_landlord_tenant_key: { propertyId, landlordId, tenantId } } })`) ‚Äî DB index name is `conversations_property_id_landlord_id_tenant_id_key` (Prisma default, underscore-joined column names). US-012 will reuse this exact client key for the upsert.
- Separate per-user lookup indexes: `@@index([landlordId], name: "conversations_landlord_idx")` + `@@index([tenantId], name: "conversations_tenant_idx")`. These support "list all conversations for user X" without having to filter by the full compound key ‚Äî future list endpoints are pre-optimized.
- Inverse relations added:
  - On `User`: `conversationsAsLandlord Conversation[] @relation("LandlordConversations")` + `conversationsAsTenant Conversation[] @relation("TenantConversations")` ‚Äî two distinct names because the same user row can be both a landlord (for some conversations) and a tenant (for others). Pattern mirrors `contractsAsTenant`/`contractsAsLandlord` and `visitsAsTenant`/`visitsAsLandlord` already in the model.
  - On `Property`: `conversations Conversation[]` ‚Äî simple inverse, matches `contracts` / `expenses` / `rentalPayments`.
- `onDelete: Cascade` chosen for all three FKs. Conversation is a chat thread ‚Äî if any participant or the property is deleted, the thread is orphaned/meaningless. Same convention as `Favorite` (Cascade on both user+property). This is a deviation from the Contract pattern (Restrict on both), which makes sense there because contracts are durable business records; conversations are not. Documented in the doc-comment above the model.

- **Learnings for future iterations:**
  - For chat-thread-like models that are ephemeral (no durable business record value), use `onDelete: Cascade` on all participant FKs rather than the Restrict default used for Contract/Visit/Proposal. The test is: "if this user / this property vanishes, does the row still make sense?" If no ‚Üí Cascade. If yes ‚Üí Restrict. Conversation/Favorite/Notification are the Cascade cluster; Contract/Visit/Proposal are the Restrict cluster.

  - Prisma's `Conversation` type export alphabetizes after `Contract` ‚Äî don't import both in the same file with a wildcard or you'll get module bloat. Prefer named imports (`import { Conversation } from '@prisma/client'`) ‚Äî same pattern as `RentalPayment`/`RentalPaymentStatus` in US-009.
