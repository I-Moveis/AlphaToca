## Codebase Patterns
- Project uses npm (package-lock.json); run `npx tsc --noEmit` for typecheck and `npm test` (vitest) for tests.
- Source lives under `src/` with subdirs `middlewares/`, `services/`, `routes/`, `controllers/`, `utils/`, `config/`; tests in `tests/`.
- Prisma is the ORM (`@prisma/client` 6.4.1). Use `prisma.$transaction` for atomic multi-table writes.
- Validation uses Zod; schemas live in `src/utils/` (e.g., `propertyValidation.ts`).
- `npm install <pkg>` defaults to the absolute latest tag — pin to a version range (e.g. `@^1.4.5-lts.2`) when the PRD asks for a specific major line.
- `npm install <pkg>` without `--save-dev` puts the package in `dependencies`, even for `@types/*`. Always pass `--save-dev` for type packages.
- Test suite has 16 pre-existing failures from Auth0 → Firebase migration (tests reference removed `userService.upsertUserFromAuth0`). Not caused by recent stories; verify regressions by running `git stash -u && npm test` vs. current.
- API error response shape across the codebase: `{ status: number, code: string, messages: [{ message: string }] }` (see `src/middlewares/authMiddleware.ts`). Stay consistent when adding new error branches.
- Uploads directory lives at project root. `src/app.ts` uses `path.join(__dirname, '../uploads')`; services under `src/services/` should use `path.join(__dirname, '../../uploads')` so both ts-node (from `src/`) and compiled (from `dist/`) resolve to the same project-root `uploads/`.
- `z.coerce.boolean()` is UNSAFE for multipart request bodies — it treats any non-empty string as truthy, so `"false"` becomes `true`. Always use a `z.preprocess` helper that explicitly maps `'true'`→true and `'false'`→false (pattern already in `src/utils/searchValidation.ts`).
- `z.coerce.number()` is the right way to accept both multipart strings and JSON numbers in the same schema. Compose with `.int()`, `.nonnegative()`, `.positive()` as needed — those run AFTER coercion.
- For Prisma `Decimal(10,2)` columns validated via Zod, pattern is `z.coerce.number().<constraint>.transform((v) => new Prisma.Decimal(v))`. Passing `Prisma.Decimal` to `prisma.property.create` works — Prisma also accepts number/string but Decimal avoids floating-point drift.
- When a Zod schema uses `.parse(req.body)`, Zod's default object mode is "strip" — unknown keys are silently dropped from the output. Adding new OPTIONAL fields to an existing schema never breaks existing callers.
- `prisma.$transaction(async (tx) => ...)` does NOT roll back non-DB side-effects (filesystem writes, external API calls). Wrap the call in an outer try/catch, track what state was created, and compensate in the catch. Inside the callback, use `tx`, never `prisma` — mixing them bypasses the transaction.
- `express.static` auto-blocks path traversal (both literal `..` and URL-encoded forms) — returns 404 for anything resolving outside the static root. No extra guards needed when serving a fixed project-root directory.
- Before implementing a PRD story, grep current code first — an AC may already be satisfied by an earlier unrelated PR. If so, verify behavior, document the deviation (if any), mark `passes:true` — DON'T regress existing code to match the PRD's literal wording.
- Global error handler lives at `src/middlewares/errorHandler.ts` and is the ONLY place to add new error-class-to-API-shape mappings. It's registered LAST in `src/app.ts` via `app.use(errorHandler)`. When adding a new branch, place it BEFORE the `SyntaxError` catch-all (which is itself before the generic fallback) so more-specific matches win. Stack traces must be logged via `logger.error` (pino) but NEVER included in the JSON response body.
- `multer.MulterError` has a `.code` string field — branch on `err instanceof multer.MulterError` then switch/lookup `err.code`. Known codes: `LIMIT_FILE_SIZE`, `LIMIT_FILE_COUNT`, `LIMIT_UNEXPECTED_FILE`, `LIMIT_PART_COUNT`, `LIMIT_FIELD_KEY`, `LIMIT_FIELD_VALUE`, `LIMIT_FIELD_COUNT`. Always include a safe fallback (e.g. `UPLOAD_ERROR`) for unmapped codes so future multer versions don't 500.
- Swagger is configured in `src/config/swagger.ts` (components + swagger-jsdoc options) and mounted at `/api-docs` from `src/server.ts` via `setupSwagger(app)` — NOT from `src/app.ts`. If you add a new reusable component schema, add it under `components.schemas` in swagger.ts; JSDoc `@swagger` blocks inside `src/routes/*.ts` can then `$ref` it. Use `allOf: [$ref Base, { properties: { extraField: ... } }]` when you need to extend a referenced schema for a specific response (e.g., response shape = Property + populated `images` array).
- `swagger-jsdoc` is permissive — an invalid JSDoc block fails SILENTLY (drops the path from the spec) rather than throwing. After editing a `@swagger` block, verify by running a throwaway smoke: `swaggerJsdoc({ apis: ['./src/routes/*.ts'] })` and asserting your path/method survived. Better: supertest GET `/api-docs/swagger-ui-init.js` and grep for the path — confirms swagger-ui-express successfully loaded the spec.
- When integration-testing a route that depends on the full app pipeline (middleware + errorHandler + validation), import `../src/app` and drive it with `supertest(app)`. Stub Firebase env vars BEFORE the import (see `tests/seedUuidRegression.test.ts`) since `src/config/firebase.ts` runs at module load. Mock the service layer (not Prisma) via `vi.mock('../src/services/...', async (importOriginal) => { const actual = await importOriginal(); return { ...actual, service: { ...actual.service, method: mockFn } }; })` — that preserves untouched methods while letting tests control the one under exercise. - Files changed: `src/services/propertyService.ts` (only).
- **Learnings for future iterations:**
  - Prisma's interactive transaction callback (`prisma.$transaction(async (tx) => {...})`) is the right tool when you need non-DB side-effects (filesystem writes) interleaved with DB writes — BUT the side-effects are NOT automatically rolled back. You must do cleanup yourself in an outer catch. The callback receiving `tx` gives you a scoped client; all DB calls inside should use `tx`, never `prisma` (mixing them would bypass the transaction).
  - `findUniqueOrThrow` is preferable to `findUnique` as the final statement in a create-transaction — the record must exist (we just created it in the same transaction), so `null` would indicate a serious invariant violation rather than a normal "not found" case. Throwing is the correct signal.

  - Ordering of state-capture variables matters: `createdPropertyId = property.id` must come BEFORE `await savePropertyImages(...)` so that a filesystem failure still leaves `createdPropertyId` set — but `savedUrls = saved.map(...)` must come AFTER the await resolves, so a throw from savePropertyImages keeps `savedUrls = []` (savePropertyImages handles its own cleanup in that case). This ordering avoids double-cleanup bugs.
  - Cleanup errors in the catch block should be swallowed (or logged at error level) and the ORIGINAL error re-thrown. `cleanupPropertyImages(...).catch(...)` achieves this without a nested try/catch.
