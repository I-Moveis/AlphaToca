## Codebase Patterns

The plain object-literal mock form (see `conversationResolve.test.ts`) shadows `requireRole` with a pass-through and makes 403 untestable. See `tests/adminSupportTicketsList.test.ts` for the importOriginal pattern.
- Role-gated route mounts chain middlewares at the `app.use` level rather than spreading them per route: `app.use('/api', authStack, requireRole(Role.LANDLORD), landlordRoutes)`. Routes inside the router stay flat.
- `@db.Decimal(10, 2)` columns emit `DECIMAL(10,2)` (not `NUMERIC(10,2)`) in Prisma-generated DDL. When PRD acceptance criteria call for `NUMERIC`, follow `prisma migrate diff` output instead — byte-for-byte match is the ACTUAL acceptance gate. Deserialize Decimal → number in the view with `Number(row.field)` (matches `contractService.getActiveContractByPropertyAndTenant`).
- `prisma.$queryRaw` tagged-template substitutions (`${value}`) are parameterized as prepared-statement placeholders — safe from SQL injection. For UUID parameters on uuid-typed columns, add a `::uuid` cast in the SQL (`WHERE landlord_id = ${id}::uuid`) so Postgres accepts the string binding. `COUNT(*)` returns `bigint` from Prisma raw queries — coerce with `Number(row.count)` in the TS caller.
- In service unit tests that call `$queryRaw` multiple times, mock the method with `.mockResolvedValueOnce(...)` chains in the exact order the service issues queries. `vi.clearAllMocks()` in `beforeEach` resets the chain between tests.
- Per-route role gating on `propertyRoutes` (mounted without authStack at `app.use('/api', propertyRoutes)`) uses a local `landlordAuthStack = [checkJwt, authSyncMiddleware, requireRole(Role.LANDLORD)]` spread on the route, mirroring the existing `adminAuthStack` pattern. Don't move `propertyRoutes` under the global `authStack` mount — too many routes in that file are intentionally public (getById, search, list).
- Route-specific paths MUST be registered before the `:id` param route in the router file. Express path-matching rejects `/properties/:id` for `/properties/analytics/monthly` by segment count (3 vs 2), BUT keeping specific routes above `:id` avoids surprises when anyone later adds a `:id/subroute`. Place new specific GETs right after `/properties/search`.
- HTTP tests for controllers that delegate to a service module: mock the service, not `prisma`. Example for LL-005: `vi.mock('../src/services/analyticsService', () => ({ analyticsService: { monthlySeries: mockMonthlySeries } }))`. Then assert the service was called with the controller-parsed args (e.g. Date objects, landlordId from req.localUser).
- Fire-and-forget tracking services (profileView/propertyView) all follow the same shape: try/catch wrapping a conditional dedup findFirst + create, with errors swallowed via `logger.error`. When a tracking call also mutates a sibling table (e.g. `Property.views` counter in LL-006), keep BOTH writes inside the same try block so a failure in either surfaces only in logs — the controller stays unaware.
- NOT every analytics event service is fire-and-forget. Contact-click (LL-007) is awaited and propagates errors — the HTTP endpoint `POST /:id/contact-click` is called synchronously from the UI and the caller expects a 201, so a DB failure should surface as 500, not a silently-logged no-op. Rule of thumb: service is fire-and-forget when wrapping a read (GET/details); service is awaited when the client explicitly called an event-recording endpoint.
- PUBLIC routes on `propertyRoutes` (mounted at `app.use('/api', propertyRoutes)` without any global auth) can still read `req.localUser?.id ?? null` — if a future middleware populates localUser optionally, the service gets it; otherwise null. Test the authenticated branch by building a mini express app that injects `req.localUser` via a tiny middleware then mounts the production router. See `tests/contactClickEvent.test.ts::buildAuthedApp`.
- When co-locating HTTP tests (vi.mock at top level for the service) AND direct service unit tests in the same file, `vi.doMock` inside a nested describe cannot override the top-level service mock because the module graph is already resolved. Solution: split service unit tests into a sibling `*Service.test.ts` file. Avoid trying to toggle mocks per-describe via `resetModules` — the cached imports will bite you.
- For per-property ownership-gated reads (e.g. LL-008 analytics), the guard ordering is: 401 (no `req.localUser`) → 400 (Zod params/query) → 404 (missing property, BEFORE the ownership check) → 403 (exists but caller !== landlordId). The 404-before-403 order prevents an existence oracle — swapping it lets callers probe "is this uuid a real property I don't own?" via the response code. Test BOTH the owner-404 AND the intruder-404 branches.
- Aggregating multiple independent Prisma counts in a service? Fire them in `Promise.all([...])` — the Prisma driver supports concurrent statements outside a transaction, and the total latency collapses from sum-of-queries to max-of-queries. Use this shape when the counts share no state (LL-008 did 6 counts + 1 raw group-by in parallel). Do NOT wrap them in `$transaction([...])` unless you actually need a consistent snapshot — transactions serialize.
- For daily-granularity analytics buckets, use `to_char(ts AT TIME ZONE 'UTC', 'YYYY-MM-DD')` in the `$queryRaw` group-by — the explicit `AT TIME ZONE 'UTC'` avoids drift when Postgres's session timezone differs from UTC. Zero-fill in TS via an `enumerateDays(fromInclusive, toExclusive)` walker + `Map.get(date) ?? 0` lookup, same pattern as `enumerateMonths` in `analyticsService.ts`.
- Ownership-gated route tests can run with a single app instance if the authMiddleware mock uses a header-driven switch (`Bearer landlord-owner` vs `Bearer landlord-intruder`) that remaps `req.localUser.id` to different UUIDs. This is the `rentalPaymentCurrent.test.ts` pattern — reuse it for any new owner-only route so you exercise the 200, 403, and 404 branches from the same `describe`.
- For "tenure-window" filters (LL-009), prefer a 2-step approach over a single join: (1) fetch `Contract.findMany({ where: { propertyId, tenantId }, select: { startDate, endDate } })` to enumerate the set of valid YYYY-MM periods in-process, (2) `rentalPayment.findMany({ where: { propertyId, period: { in: [...] } } })`. This avoids a complex raw-join and is safe because a tenant's contract count is O(few) per property. Multiple stints are naturally handled by iterating each contract's window and union-ing the period set.
- Path parameter naming is per-route freedom: the codebase uses `:id` for most property routes but LL-009 adopted `:propertyId` to match PRD wording. Both work; Express differentiates routes by path structure, not param name. Use whichever makes the handler call site read clearly.
- Prisma `@@map("snake_case_table")` is load-bearing — omitting it emits a PascalCase table name (e.g. `ConversationMessage`) that would diverge from the codebase's snake_case convention. Always add the explicit `@@map` when introducing a new model. Same goes for `@map("snake_case")` on individual columns (conversation_id, created_at, read_at, etc.). `prisma migrate diff` will show the final table/column names — compare those, not the Prisma model names.
- For model-only stories (no service/controller), the round-trip test pattern is: mock `prisma.<model>.{create, findUnique, findMany}`, then call each method with a realistic payload and assert the mock received the expected shape. Add a compile-time assertion for any new inverse relation (`const include: { <relation>: true } = { <relation>: true }`) — if the schema edit on the parent model is missed, typecheck fails. See `tests/conversationMessage.test.ts`.
- Role-agnostic endpoints (e.g. LL-011 inbox list) should decide counterpart identity via direct id-comparison (`conversation.landlordId === userId ? conv.tenant : conv.landlord`), NOT via `req.localUser.role`. This keeps the behavior correct if the same User ever appears as both sides in different threads (edge case, but the id-comparison costs nothing and removes the coupling). Same idea applies to any "other participant" surface in the chat stack.
- Boolean query params ingest as strings. Use `z.enum(['true','false']).optional()` + `=== 'true'` in the controller rather than `z.boolean()` — Zod won't coerce `"true"` to `true` automatically for query params, and `=== '1'` / `=== ''` silently becoming `false` is a footgun worth blocking at the schema layer (returns 400 instead of reinterpreting). Applied on `unreadOnly` for LL-011; LL-022's `hasWifi`/`hasPool` search filters should follow the same pattern unless `z.preprocess` is already established in the file.
- For endpoint-level empty-result assertions, return `[]` from the service WITHOUT issuing the secondary aggregation query when the primary `findMany` returns no rows. Test this with `mockGroupBy.not.toHaveBeenCalled()` — the saved round-trip matters on a cold Supabase connection and it also makes the test suite fail loudly if someone accidentally reorders the logic so the aggregation runs first.
- Prisma's `_count` with `where` filter is 4.3+, but the `groupBy` path also works cleanly for filtered-counts-per-parent and mocks more predictably. LL-011 used `prisma.conversationMessage.groupBy({ by: ['conversationId'], where: {...}, _count: { _all: true } })` — the result shape is `[{ conversationId, _count: { _all: n } }]`. This pattern extends to any "N unread per conversation", "N pending per property", etc. Future stories: prefer `groupBy` over filtered `_count` for test-mocking ergonomics.
- "Existence-hiding 404" for membership-gated resources (chat threads, DM rooms): when the caller is authenticated but NOT a participant, return 404 — NOT 403. Distinguishing 404-missing vs 403-exists-but-forbidden gives an attacker an oracle to enumerate real ids. LL-012 collapsed both branches into the same 404 handler via a single `findUnique` + `(landlordId !== userId && tenantId !== userId)` check. Apply the same rule to LL-013 (POST message) and LL-015 (mark-read).
- Cursor-based "page anterior" pagination with `take + orderBy DESC + reverse` is the cleanest SQL-native shape when the public contract wants ASC pages. The `before` cursor is resolved to a `createdAt` via `findUnique({where: {id: before}, select: {conversationId, createdAt}})`; reject forged cursors (wrong conversationId → return `[]`) to close the oracle. Cursor-as-id is simpler to pass through query strings than cursor-as-timestamp and lets the server choose what to index on (createdAt here; could be any orderable column in the future).
- Read-receipt side effect inside a paginated GET: after the `findMany`, filter `rows` to `authorId !== userId && readAt === null`, batch the ids into `updateMany({where: {id: {in: [...]}}, data: {readAt: now}})`. updateMany doesn't return rows — preserve the ids list from the already-fetched rows and use it to (a) patch the response payload's readAt to now, (b) return to the controller for LL-014's socket emit. Skip the updateMany call entirely when the filtered list is empty — saves a round trip on "all already read" pages.
- Socket emit services (chatSocketService, conversationSocketService) follow a shared shape: a tiny `safeEmit(room, event, data, ctx)` helper wraps `getIO().to(room).emit(event, data)` in try/catch + `logger.error` so a socket-engine failure never propagates to the HTTP caller (the DB write already succeeded; sockets are best-effort). Per-participant broadcasts use room convention `user:${userId}` (joined at connection time in `src/config/socket.ts::io.on('connection')`). DO NOT fan out user-to-user chat events to `provider:all` — that room is reserved for WhatsApp/RAG flows that admins legitimately observe (per PRD §8 Q5). Tests mock `'../src/config/socket'` with `{ getIO: vi.fn().mockReturnValue({ to: mockTo }) }` where `mockTo.mockReturnValue({ emit: mockEmit })` — assert rooms/events/payloads on `mockTo` + `mockEmit`.
- When a controller invokes a fire-and-forget emitter AFTER a successful DB write: place the emit call AFTER the mutation, BEFORE the `res.json(...)`. Emit errors are already swallowed by safeEmit, so they never block the response; placing the emit before the persistence would risk publishing a ghost event for a mutation that later failed. For read-receipt side effects (LL-012), emit ONLY when `markedReadIds.length > 0` — empty ids mean no state transitioned, and emitting a zero-length event is pure noise. For new-message broadcasts (LL-013), symmetric fanout to both participants (including the author) simplifies multi-device: the author's second device also receives the echo and can hydrate its local thread without a re-fetch.
- For "capture updated ids" on a bulk UPDATE, Prisma's `updateMany` only returns `{ count }` — no ids. Two options: (a) SELECT → updateMany (the LL-012 pattern, with a small race window if two requests hit the same unread rows); (b) `$queryRaw` `UPDATE ... RETURNING id` (atomic single round trip). When the PRD explicitly calls for RETURNING (LL-015), use option (b) — it also nukes the race window because there's no intermediate SELECT. HTTP tests mock the service, so the RETURNING mechanics are invisible at the handler layer; service-level tests assert the SQL shape (`expect(joined).toContain('RETURNING "id"')`) and the value binding order (tagged-template `call.slice(1)` captures prepared-statement parameters in order).
- HTTP tests that want to ASSERT a socket emit (not just tolerate it) should mock `'../src/services/conversationSocketService'` directly. Without the mock, safeEmit swallows the `getIO` "Server not initialized" throw via `logger.error` — harmless for correctness but means the test can't observe call args. The mock pattern (see `tests/conversationMarkRead.test.ts::vi.mock`) exposes both methods (emitMessagesRead + emitNewMessage) as vi.fn() so the same file can assert "was called" AND "was NOT called on the 0-ids branch". LL-012's and LL-013's tests chose the silent route (scope discipline); any test that wants to PROVE the socket contract should pick the explicit mock.
- When extending a projected view (typed service return with explicit `select` + mapper), the new field needs THREE edit sites: (1) the TS view type, (2) the Prisma `select: { ... }`, (3) the return mapper. Missing any is silent — Prisma's generated `ContractGetPayload<...>` is structurally wider than your mapper. Lock the shape with a full `.toEqual({...})` on the HTTP response body in at least one test (see `tests/contractByPropertyTenant.test.ts::'landlord owner'`) so shrinking the response surface fails loudly. Endpoints that return the FULL Prisma row via `include: { ... }` (no explicit select/mapper) auto-pick up new schema fields after `prisma generate` — no code change needed.
- For endpoints with a single role-based authorization check (e.g. "only the landlord can..."), factor the auth check into the SERVICE by throwing a domain error (`ContractError(403)` / `PropertyError(403)`). The controller stays compact: `const result = await service(id, callerId, input); res.json(result)` — errors fall out through `handleContractError` or equivalent. The shared-check cases (LL-012/LL-013/LL-015 participation across 3+ endpoints) still own the check at the controller layer to avoid duplicating the `findUnique` in every service call. Rule of thumb: unique-to-one = service layer; shared-across-many = controller layer.
- Additive enum extensions use `ALTER TYPE "Foo" ADD VALUE '<NEW>'` — one statement per new value, NO DROP/CREATE TYPE. Keep the migration enum-only (don't mix with table DDL in the same migration). All `z.nativeEnum(Foo)` validators + every `include: ...` read path auto-inherit the new surface after `prisma generate` — typical story footprint is schema + migration + tests only.
- Zod enum validators in this codebase MUST use `z.nativeEnum(PrismaEnum)` (not `z.enum(['A','B',...])`). `z.nativeEnum` stays in sync with the generated Prisma client; `z.enum` with a hardcoded string list becomes a stale allowlist that rejects newly-added enum values. Before any enum-extension story, grep `APARTMENT.*HOUSE.*STUDIO` and `ACTIVE.*PENDING.*TERMINATED` (or the appropriate marker triple) in `src/` — any hit is a stale `z.enum` allowlist that needs flipping. Swagger JSDoc `enum: [...]` lists are duplicated separately and still need manual updates — grep `enum: \[APARTMENT` to find them.

## 2026-05-07 - LL-004 — analyticsService.monthlySeries helper
- Added `src/services/analyticsService.ts` exposing `monthlySeries(landlordId, from, to): Promise<MonthlySeries>`.
  - `months`: inclusive YYYY-MM list produced by an in-process `enumerateMonths(from, to)` walker (UTC-aligned with `currentPeriod` in rentalPaymentService) — handles year-boundary crossings, single-month ranges, and `from > to` (returns empty arrays, zero queries issued).
  - `rentals[m]`: `$queryRaw` over `contracts` grouped by `to_char(start_date, 'YYYY-MM')`, filtered by `landlord_id = ${landlordId}::uuid AND start_date >= ${rangeStart} AND start_date < ${rangeEndExclusive}`.
  - `newTenants[m]`: subquery `SELECT tenant_id, MIN(start_date) AS first_start FROM contracts WHERE landlord_id = ${landlordId}::uuid GROUP BY tenant_id` wrapped in an outer `WHERE first_start >= rangeStart AND first_start < rangeEndExclusive GROUP BY to_char(first_start, 'YYYY-MM')`. Matches PRD AC: only the tenant's FIRST contract with this landlord counts.
  - `monthlyRevenue[m]`: joins `rental_payments` ↔ `properties`, filters by `p.landlord_id = ${landlordId}::uuid AND rp.status = 'PAID' AND rp.period BETWEEN fromPeriod AND toPeriod`, `COALESCE(SUM(rp.amount), 0) AS revenue`. Revenue bucket is 0 for months with no paid rows even when Postgres never emits a row for that period (zero-fill via `Map.get(m) ?? 0` in TS).

Per-value `it(...)` blocks would bloat the file 4x with identical logic. `it.each` from Vitest supports an array-of-scalars variant (`it.each([A, B, C])('label %s', (val) => ...)`) that's perfect for enum coverage.

- Added `Property.hasWifi Boolean @default(false) @map("has_wifi")` and `Property.hasPool Boolean @default(false) @map("has_pool")` to `prisma/schema.prisma`, placed immediately after the existing `isFurnished` / `petsAllowed` boolean amenity flags (same logical grouping).
- Hand-authored `prisma/migrations/20260508050000_add_property_amenities/migration.sql`: single `ALTER TABLE "properties" ADD COLUMN "has_wifi" BOOLEAN NOT NULL DEFAULT false, ADD COLUMN "has_pool" BOOLEAN NOT NULL DEFAULT false;`. Column declarations match the `--from-empty` diff byte-for-byte (both emit `"has_wifi" BOOLEAN NOT NULL DEFAULT false` and `"has_pool" BOOLEAN NOT NULL DEFAULT false`). No backfill needed — every existing row's absence of the amenity is correctly represented by the `DEFAULT false`.
- `npx prisma validate` + `npx prisma generate` both succeed. The regenerated client exposes `hasWifi: boolean` + `hasPool: boolean` on every Property-returning query path automatically.
- No controller / service / Zod edits in this story per PRD scope. LL-022 owns write-path support (POST/PUT JSON + multipart) and search filter integration. Every Property-returning endpoint (GET /search, GET /:id, POST, PUT) returns the full Prisma row via `include`, so the new fields surface in responses as soon as prisma generate lands — dashboards can start reading them without waiting for LL-022's write path.
- Files changed: `prisma/schema.prisma`, `prisma/migrations/20260508050000_add_property_amenities/migration.sql` (new), `tests/propertyAmenitiesColumns.test.ts` (new — 5 tests).
- Tests: 5/5 new pass — both-true round-trip, explicit both-false round-trip, defaults-applied-when-omitted (asserts caller's `data` payload does NOT carry either key, proving the `false` came from the DB-level DEFAULT), mixed flags (hasWifi=true / hasPool=false), migration DDL substring + negative-shape assertions (no DROP COLUMN / DROP TABLE).
- `npx tsc --noEmit` clean. Full suite: 609/630 pass — same 21 pre-existing failures (auth/userSync/ingestKnowledge/rag*/visitValidation); net delta is +5 new passing, zero regressions.
- **Learnings for future iterations:**
  - For a purely additive boolean-column schema change with `@default(false)`, the migration is the ONE statement `ALTER TABLE ... ADD COLUMN ... BOOLEAN NOT NULL DEFAULT false, ADD COLUMN ...` — comma-joined, no backfill UPDATE. Postgres atomically adds both columns in the same DDL statement and applies the default to every existing row. Cleaner than two separate ADD COLUMN statements for thematically-linked flags (amenities pair here).
  - When the PRD's "Tests pass" AC mentions "default false when omitted," mocking a Prisma create that returns the default requires a two-part assertion: (a) the mock returns `{ hasWifi: false, hasPool: false }` as the DB would, AND (b) the test verifies the CALLER's `data` payload did NOT include either key. Without (b), the test would silently pass even if the application layer were injecting `false` manually — missing the actual contract that the DEFAULT comes from the DB schema, not the app code. Pattern: `expect(call.data).not.toHaveProperty('hasWifi')`.
  - When adding multiple boolean flags in the same story, testing each combination (both true, both false, mixed, default) with separate it() blocks keeps the output readable and catches the rare bug where one flag's mapper is wired and the other's isn't. Parameterized `it.each` is overkill for 2 flags but would shine if this were hasWifi + hasPool + hasGym + hasParking.
  - `prisma migrate diff --from-empty --to-schema-datamodel` emits CREATE TABLE with the two columns inline. The ADD COLUMN migration matches byte-for-byte on the column declaration substring (`"has_wifi" BOOLEAN NOT NULL DEFAULT false`). When asserting this in tests, normalize whitespace first (`sql.replace(/\s+/g, ' ')`) so the hand-authored multi-line `ADD COLUMN     "has_wifi" ...` shape passes the regex.
  - Next migration timestamp ceiling is now `>= 20260508050001`. LL-022 (amenity support in create/update/search) is HTTP-only — no migration needed; it depends only on LL-021's schema already being in place. After LL-022, the epic is complete.

## 2026-05-07 - LL-022 — Amenity support in create/update/search
- Extended `createPropertySchema` + `updatePropertySchema` in `src/utils/propertyValidation.ts` with optional `hasWifi` + `hasPool` booleans using the shared `multipartBoolean` preprocessor — the JSON path passes real booleans through and the multipart path coerces `'true'`/`'false'` strings into booleans while rejecting anything else at the Zod layer (400 VALIDATION_ERROR).
- Extended `propertySearchSchema` in `src/utils/searchValidation.ts` with `hasWifi` + `hasPool` via the existing `stringToBoolean` transformer — omitted query params resolve to `undefined` (no filter), `'true'`/`'false'` produce booleans, other inputs silently resolve to `undefined` (matches existing isFurnished/petsAllowed behavior in the same file).
- Wired both filters into `propertyService.searchProperties` (Prisma `where` path) with `...(hasWifi !== undefined && { hasWifi })` + `...(hasPool !== undefined && { hasPool })`. The raw-SQL branch (location/radius search) intentionally does NOT carry the amenity filters — consistent with the pre-existing limitation of isFurnished/petsAllowed/nearSubway on that path.
- Controllers didn't need changes: `create` forwards `validatedData` to `propertyService.createProperty` which spreads `data` into `prisma.property.create`; `update` forwards `validatedData` to `propertyService.updateProperty` which splits off `photosToRemove` and spreads the rest as `scalarData` into `prisma.property.update`. Both paths auto-pick up the new fields from the Zod schema extension.
- Swagger JSDoc updated at 4 call sites: POST /properties JSON + multipart request bodies (propertyRoutes.ts), PUT /properties/:id multipart requestBody (noted string coercion semantics), GET /properties/search query params (both flags documented), plus the shared `Property` component schema in `src/config/swagger.ts` (both fields with `default: false` and LL-021 context notes).
- Files changed: `src/utils/propertyValidation.ts`, `src/utils/searchValidation.ts`, `src/services/propertyService.ts`, `src/routes/propertyRoutes.ts`, `src/config/swagger.ts`, `tests/propertyAmenitiesFlow.test.ts` (new — 11 tests).
- Tests: 11/11 new pass. POST JSON (both flags true forwarded + 201 echo, both flags omitted leaves data without keys), PUT JSON (both flags updated and echoed), PUT multipart (`'true'`/`'false'` coerced to booleans + forwarded; `'yep'` rejected 400 VALIDATION_ERROR), GET /search (hasWifi=true only, hasPool=true only, AND semantics, hasWifi=false negative filter, omitted flags produce no amenity keys in `where`), Zod schema parse sanity for string coercion.
- `npx tsc --noEmit` clean. Full suite: 619/641 pass — same 22 pre-existing failures (auth/userSync/ingestKnowledge/rag*/visitValidation + the occasional visitService.cancelVisit parallel-run flake); net delta is +11 new passing, zero regressions. Adjacent property tests co-run: 47/47 pass (amenitiesColumns, searchExtendedTypes, updateMultipart, photosRemove, propertyStatus, currentTenant, photoUpload).
- **Learnings for future iterations:**
  - When extending `createPropertySchema` + `updatePropertySchema` simultaneously, reuse the same `multipartBoolean` preprocessor that's already defined at the top of `propertyValidation.ts`. This keeps the JSON-vs-multipart coercion contract uniform across create and update paths — no surprise where one path accepts `'true'` strings and the other doesn't. The existing isFurnished/petsAllowed in `createPropertySchema` set the precedent; LL-022 follows it without introducing a new helper.
  - For search filter extensions, check whether the existing raw-SQL (geo-search) branch already passes peers of the new flag. In this codebase, the raw-SQL branch under `lat`/`lng`/`radius` does NOT carry isFurnished/petsAllowed/nearSubway/isFeatured — so hasWifi/hasPool also don't need to go there. If a future story decides geo-search must honor amenities, it'll be a separate scope-increase (add all four amenity filters as SQL AND clauses in the raw SQL path — keep them symmetric or the UX diverges between "near me" vs. "full-text search").
  - End-to-end search testing pattern: mock `prisma.property.count` + `prisma.property.findMany` at the `../src/config/db` layer and let the real controller+service stack run. The `filterRows(where)` helper in the test consults `args.where.hasWifi` / `args.where.hasPool` and returns a subset of a 4-row STORE — same semantics as a live DB seeded with one property per amenity combination. This lets the single test file exercise positive filter, negative filter, AND semantics, and "omitted flags yield no filter key" in ONE describe block.
  - The existing `propertyUpdateMultipart.test.ts` header-switch auth pattern (`Bearer landlord-owner` vs `Bearer landlord-intruder` remapping `req.localUser.id`) works cleanly for PUT flows that need BOTH a 200 owner path AND a hypothetical 403 intruder path in the same describe. LL-022's test doesn't exercise the 403 (that's already locked in by propertyUpdateMultipart.test.ts), so it stays owner-only — but reuses the exact middleware mock shape so future amenity-related ownership tests can extend without refactor.
  - `multipartBoolean`'s preprocessor returns `val` unchanged when it's neither `'true'` nor `'false'` — so `z.boolean()` then rejects with a `VALIDATION_ERROR`. This is the correct guard: it BLOCKS silent misinterpretation of inputs like `'yep'`, `'1'`, `'on'`. If the product ever asks to support a third truthy form, extend the preprocess before the `z.boolean()` step rather than loosening the downstream schema — the error surface is where bad inputs should die.
  - The `stringToBoolean` transformer in `searchValidation.ts` silently resolves unknown values to `undefined` (no filter), which DIFFERS from `multipartBoolean` in `propertyValidation.ts` (which rejects with 400). Rationale: search query params are user-land URL hacks and a bad value shouldn't 400 the entire search — it just means "no filter for that flag, pass through everything." This is the existing pattern the Codebase Patterns section warns about (z.enum + `=== 'true'` vs z.preprocess); LL-022 preserves it rather than diverging. For create/update, 400 is the right outcome — the client is POSTing a mutation and bad input must halt.
  - Migration timestamp ceiling unchanged (`>= 20260508050001`). LL-022 had no schema changes — LL-021 already landed both columns. With this story, the epic is complete (all 22 stories now passing).
