## Codebase Patterns
- Prisma client singleton lives in `src/config/db.ts` (reuse via `import prisma from "../config/db"`) — do NOT instantiate a second `PrismaClient`.
- Tests use `vitest` (config at `vitest.config.ts`). Test files live in `tests/` at repo root, not colocated.
- `npx tsc --noEmit` should be clean end-to-end (no known noise). If a new error appears, it came from your changes — do NOT treat it as pre-existing. (Prior `seed-knowledge.ts` error was fixed in US-003 by switching to `@langchain/textsplitters`.)
- `KnowledgeDocument.embedding` is `Unsupported("vector(1536)")` in Prisma — writes/reads must use `$executeRawUnsafe` / `$queryRawUnsafe` with `::vector` casts. Never interpolate user input.
- `langchain@^1.3.3` is the v1 API — use LCEL `RunnableSequence` / `ChatPromptTemplate`, NOT the deprecated `ConversationalRetrievalChain`.
- Project uses `dotenv` for secrets; fail-fast on missing keys at boot rather than returning `undefined` deep in a call chain.
- `.env` `DATABASE_URL` points to Supabase (production) — NEVER run `prisma migrate dev` with default env. Use local Docker DB (`alphatoca_db` on :5433, creds `admin`/`admin_pwd`) via inline env overrides.
- `prisma migrate dev` refuses to run non-interactively in this harness. Use this two-step workaround: (1) `prisma migrate diff --from-migrations ./prisma/migrations --to-schema-datamodel ./prisma/schema.prisma --shadow-database-url <local_shadow> --script` to emit SQL; (2) write it to `prisma/migrations/<TS>_<name>/migration.sql`; (3) `prisma migrate deploy` with local env overrides to apply.
- Shadow DB (`alphatoca_shadow`) exists on the local Docker postgres for migration diffs; pgvector extension already enabled on both local DBs.
- `KnowledgeDocument` now has ingestion-tracking fields: `sourcePath`, `chunkIndex`, `contentHash`, `createdAt`, `updatedAt`. Compound unique index on `(sourcePath, chunkIndex)` — use this for upserts. Index on `contentHash` for hash-skip lookups.
- The existing `prisma/seed.ts` + `prisma/generators.ts` insert `KnowledgeDocument` rows via `$executeRawUnsafe` without the new NOT NULL columns — they will fail at runtime post-migration. US-003 did NOT fix this (the real RAG ingestion path is `npm run ingest:knowledge`); the generator path is still broken. If anyone runs `npm run seed`, update `prisma/generators.ts::generateKnowledgeDocuments` to supply `source_path` / `chunk_index` / `content_hash` / `updated_at`, then match in `prisma/seed.ts`.
- Text splitting is served by `@langchain/textsplitters` (separate package, added as a dep in US-003). `langchain@^1.3.3` no longer re-exports `RecursiveCharacterTextSplitter` from `langchain/text_splitter`. Import from `"@langchain/textsplitters"` everywhere.
- Ingestion script pattern: keep a pure `runIngestion(deps)` function (deps object with `{ prisma, embedder, splitter, listMarkdownFiles, readFile }`) + a thin CLI wrapper that constructs real `OpenAIEmbeddings` / `RecursiveCharacterTextSplitter` / prisma singleton. Makes the hash-skip / insert / update / delete logic trivial to unit-test with mocked deps, and mirrors how US-004+ retriever/chain services should be structured.
- Vector literals for pgvector go in as a string of the form `"[v1,v2,...]"` passed to `$executeRawUnsafe` with `$N::vector` cast — never interpolate the array directly into the SQL string.
- Retrieval uses `$queryRawUnsafe` with the cosine-distance operator `embedding <=> $1::vector`. Compute similarity as `1 - distance` (pgvector returns distance, NOT similarity). `score >= SIMILARITY_THRESHOLD` is the gate for "trust the LLM answer" vs. "hand off to a human".
- Services that need LLM/embedding clients should cache the constructed client in a module-level `let defaultDepsCache = null` + `getDefaultDeps()` helper. That way the module can be imported in tests without requiring `OPENAI_API_KEY`, but production callers still get a warm singleton.
- Prisma raw returns of numeric columns (e.g. pgvector distance) can come back as JS `number` OR as `string` depending on the driver — always coerce with `typeof x === "string" ? parseFloat(x) : x` at the boundary.
- To load the "last N" messages ordered ASC in a single call: use `orderBy: { timestamp: "desc" }, take: N` then `.slice().reverse()` in JS. `orderBy: asc, take: N` loads the FIRST N (oldest), not the most recent.
- `@langchain/core` message classes (`HumanMessage`, `AIMessage`, `SystemMessage`) have a `.content` field that can be `string` OR an array of content blocks (e.g. `[{ type: "text", text: "..." }, ...]`) — always normalize LLM output through a helper that handles both shapes before returning user-facing strings.
- Business-plan tone rules live in `documentation/buisness_plan.md` §6 — any chatbot-facing prompt should encode: profissional/acolhedor/confiável, base-apenas-no-contexto, handoff em litígios/exceções, PT-BR, respostas breves para WhatsApp.

  - Ordering returned by the SQL query is preserved and scores are descending (since distance asc).
