## Codebase Patterns
- Prisma enum names are globally unique in a schema — you cannot define two enums with the same name. `PaymentStatus` is already taken by `TenantPayment` (PENDING|PAID|OVERDUE|CANCELLED); the monthly rent status enum lives under `RentalPaymentStatus` (AWAITING|PAID|LATE) on `RentalPayment`. If a PRD asks for an enum name that's already bound, pick a namespaced alternative (`<Model>Status`) and document the deviation — don't rename the incumbent.
- Hand-authored migration verification: before committing a hand-authored `prisma/migrations/<stamp>_<name>/migration.sql`, run `npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script` and grep for your table name. The generated DDL for the new table/index/FK should match the hand-authored file byte-for-byte (ENUM CREATE order, VARCHAR(n) vs TEXT, DEFAULT 'X', index/constraint names, CASCADE vs SET NULL on FK). `prisma migrate diff` is read-only and safe to run against any URL — including the live Supabase pooler — since it operates on the schema file, not the DB.
- Relation onDelete conventions in this schema: child-of-Property tables use `onDelete: Cascade` (property_images, favorites, property_expenses, maintenance_logs, notifications, rental_payments) so a property delete takes the subgraph with it. User-is-actor FKs (e.g., `RentalPayment.updatedBy`) use `onDelete: SET NULL` so deleting a user preserves historical rows with a null actor. Avoid `Restrict` unless the business rule actively forbids deletion (contracts use Restrict because a pending contract should block landlord/property/tenant deletion).
- Seed IDs: random seeds use `crypto.randomUUID()` (not `faker.string.uuid()`); stable demo IDs live in `prisma/demoIds.ts` as canonical UUIDv4 constants — don't replace those constants with `randomUUID()` calls because they're referenced by tests and Firebase UID mappings.
- UUID validation: routes use `z.string().uuid()`; any seed id you introduce must match `/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i`.
- Test runner: `npx vitest run` (vitest config in `vitest.config.ts`). Many pre-existing tests fail due to an unfinished Auth0→Firebase migration (`userService.upsertUserFromAuth0` missing, `tests/auth.test.ts`, `tests/ingestKnowledge.test.ts`) — treat those as baseline noise, not regressions.
- Grep scope for "seed" lookups: include `prisma/` (generators.ts, demoData.ts, demoIds.ts, seed.ts) — seeds are not under `src/` in this repo.
- Typecheck: `npx tsc --noEmit` — must pass before commit.
- `.env` DATABASE_URL points at the live Supabase project — DO NOT run `prisma migrate dev/deploy/reset` locally; hand-author `prisma/migrations/<stamp>_<name>/migration.sql` instead and verify with `npx prisma validate` + `npx prisma generate`. Reviewer runs the actual `migrate reset --force` on a disposable dev DB before landing.
- Swagger is split: `src/config/swagger.ts` owns component schemas (e.g., `Property`); per-route JSDoc blocks in `src/routes/*.ts` own request/response shapes — when an enum changes, both files need an edit.
- Prisma enum value rename: use `ALTER TYPE "X" RENAME VALUE 'OLD' TO 'NEW';` — it rewrites the catalog label in place, so existing rows carry over with zero data loss (no `UPDATE` pass required).
- `propertyValidation.ts` uses `z.nativeEnum(PropertyStatus)` — it re-reads the Prisma client enum at import time, so bumping the enum only requires a `prisma generate`, not a Zod edit. Same convention for `rentalPaymentValidation.ts` → `RentalPaymentStatus` (US-010).
- Integration tests that need a "backing store" without a real DB: mount the real Express app via `supertest` and swap `propertyService` with `vi.mock('../src/services/propertyService', importOriginal)` spreading the actual module plus overriding only the methods you exercise. A `Map<id, record>` in `beforeEach` is enough to simulate create/update/get/search chains — Prisma model semantics are preserved by having the mock merge updates (`{...existing, ...data}`).
- Route-level auth: `app.use('/api', propertyRoutes)` in src/app.ts mounts WITHOUT the `authStack`, so POST/PUT/DELETE /properties are currently unauthenticated at the Express layer. Owner-only checks (US-006/US-007) will need the middleware added OR an inline `localUser`/ownership guard in the controller.
- Zod error shape: the global errorHandler (`src/middlewares/errorHandler.ts`) already converts `ZodError` into `{ status: 400, code: 'VALIDATION_ERROR', messages: [...] }` — route/controllers only need to `schema.parse(req.body)` and pass the ZodError to `next(error)`.
- `ProcessStatus` on `RentalProcess` in this schema is `TRIAGE|VISIT_SCHEDULED|CONTRACT_ANALYSIS|CLOSED` — there is NO `ACTIVE` value. When a PRD/handoff says "RentalProcess with status=ACTIVE", the semantic the codebase actually implements is the `Contract` model (ContractStatus=ACTIVE|TERMINATED|COMPLETED). `contractService.createContract` already couples an ACTIVE Contract insert with `Property.status='RENTED'` in one transaction — so Contract is the source of truth for "who is the current tenant".
- To add a relation-derived field (e.g., currentTenant) to a property read path without N+1: use `include: { contracts: { where: { status: 'ACTIVE' }, select: { tenant: { select: { id, name } } }, take: 1 } }` in Prisma findUnique/findMany (single JOIN), and for any raw `$queryRaw` path in the same service, emulate the include with ONE batched `prisma.contract.findMany({ where: { status: 'ACTIVE', propertyId: { in: ids } } })` followed by a Map join in memory — still constant queries per page.
- Mocking Prisma for service-level tests: `vi.mock('../src/config/db', () => ({ default: { model: { method: vi.fn() }, $queryRaw: vi.fn() } }))` at the TOP of the test file (before any import that pulls the service in), then cast with `(prisma.model.method as any).mockResolvedValue(...)` per test. See `tests/propertyCurrentTenant.test.ts` for the full template — works cleanly with `src/config/db`'s `export default prisma` shape.
- Mocking `prisma.$transaction(async (tx) => ...)`: the factory returns `{ default: { ..., $transaction: vi.fn(async (fn) => fn({ model: prismaMocks })) } }`. Default impl proxies the `tx` object to the same top-level mocks so callees can use either `tx.model.method` or `prisma.model.method` interchangeably in tests. To simulate a ROLLBACK on mid-tx failure, per-test override with `(prisma.$transaction as any).mockImplementationOnce(async (fn) => { try { return await fn(txProxy); } catch(err) { committed = false; throw err; } })` and assert the caller sees the throw AND the downstream writes never ran. See `tests/rentalLifecycle.test.ts` for the full template.
- Lifecycle coupling semantics in this codebase (US-005): Property.status transitions are coupled to Contract.status, NOT RentalProcess.status. The `ProcessStatus` enum has no ACTIVE value — the source of truth for "tenant is currently renting" is `Contract.status=ACTIVE`. Rules enforced atomically inside Prisma $transactions: (a) `contractService.createContract` — one ACTIVE contract max per property (409 RENTAL_PROCESS_ALREADY_ACTIVE) + Property→RENTED; (b) `contractService.updateContractStatus` — ACTIVE→terminal flips Property→AVAILABLE, terminal→ACTIVE re-checks the guard and flips Property→RENTED; (c) `rentalProcessService.updateStatus` — only releases Property when CLOSED AND property was in NEGOTIATING (i.e., no contract was ever signed); RENTED properties are left alone since contract termination is the authoritative release path.
- Content-Type-conditional multer: to let a single route accept BOTH application/json AND multipart/form-data, wrap `multer.array(...)` in a small handler that only invokes multer when `req.headers['content-type']` starts with `multipart/form-data` (case-insensitive), otherwise `next()`. Multer consumes the request stream, so running it on a JSON body would shadow `express.json()` — never apply it unconditionally. `src/middlewares/uploadMiddleware.ts#conditionalPropertyPhotoUploadHandler` is the reference for this pattern; reuse it for US-007 (photosToRemove) and US-016 (signed PDF) rather than writing a new one.
- Auth gating on the property route group: `app.use('/api', propertyRoutes)` deliberately mounts WITHOUT the global authStack — many GET paths (search, /:id, list) are public. To gate a SPECIFIC route on that router, declare `const authStack = [checkJwt, authSyncMiddleware]` INSIDE `src/routes/propertyRoutes.ts` and spread it per-route (e.g., `router.put('/properties/:id', ...authStack, uploadHandler, controller.update)`). Ownership is enforced INSIDE the controller via `req.localUser.id === property.landlordId` — this avoids the 404-vs-403 leak (404 first if the property doesn't exist, then 403 if it does but you're not the owner).
- Cover-preservation rule on image-adding updates: when new photos are uploaded alongside a PUT that keeps existing images, NEVER promote a new photo to cover unless the property has zero covers. Pattern: `const hasExistingCover = existing.images.some(img => img.isCover)` read before the tx, then in the `propertyImage.createMany` step compute `isCover: !hasExistingCover && idx === 0`. Don't reuse `savePropertyImages`' default `isCover: i === 0` because it would silently replace the existing cover.
- Multer text fields coexist with file arrays: `multer.array('photos', 20)` accepts the FILE field `photos` AND collects any repeated non-file text fields into `req.body` (single occurrence → string, 2+ → string[]). No need for `.fields()` or a second handler — the existing `propertyPhotoUploadHandler` already routes `photosToRemove` text fields into `req.body.photosToRemove`. Normalize the string-vs-string[] shape in Zod via `z.preprocess(v => typeof v === 'string' ? [v] : v, z.array(...))` so downstream code sees a uniform array.
- `PropertyError` lives in `src/services/propertyService.ts` and mirrors `ContractError`/`VisitError`/`ProposalError` — `httpStatus, code, message`. The controller pattern is `if (err instanceof PropertyError) res.status(err.httpStatus).json({ status, code, messages: [{ message }] })` right inside the try/catch, BEFORE `next(err)`. Reuse this class for any future 4xx business rule in property service (e.g., "cannot delete property with active contract"). Don't add a new error class for each story.
- Relative-path URLs in schema: `PropertyImage.url` is stored as `/uploads/<propertyId>/<file>.jpg` — NOT an absolute `https://...` URL. Zod validation for any field carrying this value must use `z.string().min(1)` (or a path regex), NOT `z.string().url()`. Check the actual stored shape via `prisma/schema.prisma` + the producer service (`propertyImageStorageService.savePropertyImages`) before picking a Zod primitive.
- Post-commit storage cleanup is fire-and-forget: after a Prisma `$transaction` that modifies rows linked to files on disk, run `await cleanupPropertyImages(id, urls).catch(err => logger.warn({err, propertyId: id}, '...'))` OUTSIDE the tx so a storage failure never rolls back committed DB state. The cleanup function already tolerates ENOENT and non-empty-dir `rmdir`, so it's safe to pass a mix of existing + already-gone URLs. Contrast with the pre-commit path inside `savePropertyImages`'s outer try/catch which DOES throw on the way up to trigger the tx rollback cleanup.
- Mocking `propertyImageStorageService` for service-level tests: `vi.mock('../src/services/propertyImageStorageService', () => ({ savePropertyImages: vi.fn(), cleanupPropertyImages: vi.fn() }))` at the top of the test file, combined with the standard `vi.mock('../src/config/db', ...)` $transaction pattern, exercises the full `updateProperty` tx path end-to-end without disk I/O. The only caveat: `Buffer` + a minimal fake `Express.Multer.File` object (fieldname/mimetype/buffer) is enough to flow through the service's type expectations since the real multer mime inspection happens upstream at the middleware layer.
- Period-computation convention (US-009+): the server-side `YYYY-MM` of the current month is computed in UTC via `new Date().getUTCFullYear()` + `getUTCMonth()+1` zero-padded. Pattern is centralized as `currentPeriod(now?: Date): string` in `src/services/rentalPaymentService.ts` — the US-010 upsert also consumes it so GET/PUT handlers agree byte-for-byte on month boundaries. Never accept a `period` from the request body/query — the PRD contract is that the client does not dictate the period (blocks retroactive edits via API).
- Prisma `upsert` on a compound-unique key: the `where` clause reuses the SAME client key used by `findUnique` (the `@@unique([...], name: "...")` value, e.g. `rental_payments_property_period_key`). The `create` payload must include all NOT-NULL columns not defaulted by Prisma (`propertyId`, `period` here — `status` is defaulted, `updatedAt` is `@updatedAt`). The `update` payload only carries the mutable fields (`status`, `updatedBy`). `@updatedAt` fires automatically on BOTH create and update — do NOT set it from the service or the mock tests would have to fake it.
- "GET returns default without persisting" pattern for compound-unique lookups: when a row may not exist yet but the UI needs a stable default shape (US-009), the service does `prisma.X.findUnique({ where: { <compound_key>: { ... } } })` and branches on `null` → returns the default in-memory instead of calling `upsert`/`create`. The write-on-read anti-pattern is easy to fall into when "GET a resource that doesn't exist" feels wrong — but the PRD contract here is explicit: AWAITING is the zero-state, not an inserted row. Keep writes in the PUT path only.
- Route protection on `propertyRoutes.ts` (added in US-006): the local `const authStack = [checkJwt, authSyncMiddleware]` is now the canonical way to gate a single route on the publicly-mounted router. Reuse it for US-009/US-010 (`/payments/current`) and US-012 (`/conversations/resolve` if that lives under the same mount) — don't reintroduce `app.use('/api', authStack, propertyRoutes)` globally or you'll break the public GET /search + GET /:id paths.
- New resource ≠ new mount pattern: when a story adds a resource that's unrelated to any existing route file (e.g. US-012 `/conversations/resolve` is not a `/properties/...` path), prefer a NEW `src/routes/<resource>Routes.ts` mounted at `app.ts` via `app.use('/api', authStack, <resource>Routes)` — not a local `authStack` spread inside the route. The local-authStack pattern (propertyRoutes.ts) exists ONLY because that router has mixed public+private paths. New resources with uniform auth go under the global `authStack` mount for less boilerplate. See `src/routes/conversationRoutes.ts` for the minimal template (1 route, no per-route auth spread).
- Query-param validation via Zod: use `schema.parse(req.query)` inside the controller. Express serializes query strings as plain `Record<string, string | string[] | ...>`, so `z.string().uuid()` works out of the box — NO preprocess needed for simple scalars. Repeated params (`?a=1&a=2`) arrive as arrays; if a param should be single-valued, either enforce `z.string()` (which will reject arrays with a ZodError → 400) or preprocess to take the first element. Missing params arrive as `undefined` — rejected by required Zod fields with a readable path.
- Server-derived identity for upsert keys: when a write path has a compound unique key and one of the components is deriveable from server state (e.g. `landlordId` from `Property.landlordId`), NEVER accept it from the request. Derive it in the controller from a prior read (`propertyService.getPropertyById(propertyId)`), then pass it to the service. Rationale: accepting it from input lets an authenticated tenant forge a thread with a fake landlord, creating orphan rows that diverge from the UI (the UI infers landlord via Property, not via the conversation row). Pattern: service signature takes the derived id as a required arg, never reads request objects.
- "Create-or-get atomic" contract via `prisma.<model>.upsert` with `update: {}`: when the business semantics are "return the existing row unchanged if present, otherwise create it", an upsert with an empty `update:` object is the right tool. The DB unique constraint converts the duplicate-INSERT race into a SELECT transparently. Tests that want to prove idempotency just assert that the service returns identical views across two calls AND that the service itself is stateless (no caching) — the DB guarantees the rest. See `src/services/conversationService.ts` for the US-012 template.
- Synthesized response fields: when an API response includes a field that is NOT a column (e.g. `messages: []` in Conversation, `currentTenant` in Property), the service layer is responsible for synthesizing it — not the controller, not the DB. Benefit: the single service-level test file covers both shape + presence, and controllers stay as thin as possible. For `Conversation.messages` specifically: the test `response body always has messages: []` guards against a future dev accidentally piping a DB column through once messages become first-class.
- Model-extend vs model-create decision (US-013): before creating a new Prisma model, grep for every existing model that shares (a) the same relation targets, (b) the same status/lifecycle field, and (c) the same invariants enforced by services. If an incumbent model checks all three, extend it with the missing columns — never create a parallel. Parallel models orphan existing service-enforced invariants (here: US-005's atomic `Property.status=RENTED ↔ Contract.status=ACTIVE`) and force every consumer to cross-reference both. Document the decision in the story notes AND at the top of the model's doc-comment so the next reader doesn't re-derive it.
- Postgres catalog-rewrite renames: `ALTER TABLE ... RENAME COLUMN` (US-013) joins `ALTER TYPE ... RENAME VALUE` (US-002) as the preferred pattern for renaming already-populated fields in this repo. Both are metadata-only — zero row rewrite, zero data loss, zero downtime — and both pair with `@map("new_name")` on the Prisma side so `prisma generate` regenerates the client with the new field name. The TypeScript compiler catches every stale caller immediately (`property 'contractUrl' does not exist on type 'Contract'`), which is the correct signal — a silent shadow field would be worse than a loud compile error.
- Nullable-without-default ADD COLUMN: when adding a column to an existing populated table where the zero-state is "no value yet" (e.g. `Contract.signedAt` before any PDF is uploaded), use `ALTER TABLE ... ADD COLUMN <name> <type>;` — NO default. Postgres ≥ 11 handles nullable-without-default as a pure metadata update (zero row touch). Adding a default forces a table rewrite (`UPDATE all rows`) which can be a minutes-long lock on large tables. Defaults belong at the application layer (Zod omission, service computation) unless there's a business reason the DB should enforce them on historical rows — which is rare.
- Swagger component schema should carry a `description` tying the model to the lifecycle endpoints that mutate it as a side-effect. Pattern: when an API response includes fields that are only populated by a specific write path (e.g. `Contract.signedAt` populated only by PUT /api/contracts/:id/signed-document), put the endpoint name + story reference in the field's `description`. Frontend integrators then discover the write path from the OpenAPI spec alone without having to read route handlers. Applied on Contract (US-013) mirroring the Property.status description pattern (US-005).
- HTTP-test and service-test mocks must NOT share a file when both target the same module. A `vi.mock('../src/services/X', ...)` at the top of a file is hoisted and applied globally — so a later `describe` block that tries to `import('../src/services/X')` to test the REAL service will get the mocked version instead, silently asserting against mock behavior. Split into two files: one that mocks the service (HTTP / controller layer) and one that mocks `src/config/db` (service layer). Each file ends up with one layer's mock and tests exactly that layer's real code.
- Service view types: when an API projection omits a field that the service still needs internally (e.g. `landlordId` is used for the authorization check in US-014 but must be omitted from the HTTP response), keep the field on the service-return type and strip it in the controller via `const { landlordId: _, ...view } = contract`. This keeps the service reusable for future callers that need the full shape, and makes the "what the wire sees" vs "what the server sees" boundary explicit at the controller boundary.
- Decimal → number in response projections: `Number(row.monthlyRent)` is the right conversion for `Decimal(10,2)` columns in this repo. The Prisma `Decimal` class stringifies to its numeric representation, so `Number()` round-trips values like `'2500.00'` → `2500` and `'3100.50'` → `3100.5` correctly for typical rent amounts. Don't use `parseFloat` (it has subtly different locale/edge-case behavior); don't send the `Decimal` object through `res.json()` directly (it serializes to a quoted string, not a JSON number, breaking frontend typed parsers).
- `res.sendFile` + error-branch `res.json()` Content-Type gotcha: `res.json()` only sets `Content-Type: application/json` when the response has NO Content-Type yet (Express source: `if (!this.get('Content-Type')) this.set('Content-Type', 'application/json')`). If you pre-set `application/pdf` via `res.setHeader` BEFORE calling sendFile, the ENOENT-branch 404 JSON envelope will be served with Content-Type: application/pdf — supertest (and real clients) parse it as binary. Fix: let `res.sendFile` derive Content-Type from the file's extension via its internal mime lookup (`.pdf` → `application/pdf`), and only set non-Content-Type headers (`Cache-Control`) explicitly upfront. Applied in `src/controllers/contractController.ts#getPdf` (US-015). Reusable for any future file-streaming endpoint with an error branch.
- Storage-backed download endpoints with dual strategy (local-stream + signed-URL-redirect): when the project today uses only local filesystem storage but may migrate to cloud storage later, write the controller to handle BOTH `/uploads/...` relative paths (stream from disk) AND `http(s)://...` absolute URLs (302 redirect) from day one. Selection is based on a `/^https?:\/\//i.test(pdfUrl)` regex. Benefits: (a) the endpoint is forward-compatible — when a future storage service produces signed URLs, no controller change is needed; (b) the Swagger doc documents both 200 and 302 branches so integrators code for both from the start. Defense-in-depth: for the local-stream branch, strip leading slashes from the stored value, resolve against UPLOADS_ROOT, and reject paths that escape the root (blocks `..` traversal in stored values). See `src/controllers/contractController.ts#getPdf` (US-015) for the template — reusable for US-016 signed PDF download and any future asset-serving endpoint.
- Multipart upload endpoints with magic-bytes validation (US-016): multer's `fileFilter` rejects based on Content-Type mime, but a client can lie (name a text file `contract.pdf` with `Content-Type: application/pdf`). For format-critical uploads (PDFs, images, archives), add a SECOND check in the controller after multer hands off: read the first N bytes of `file.buffer` and compare to the format's magic bytes (PDF → `%PDF`, PNG → `\x89PNG`, JPEG → `\xFF\xD8\xFF`, ZIP → `PK\x03\x04`). Reject with the same public code the fileFilter uses (`INVALID_FILE_TYPE`) so the frontend handles both failures with one branch. See `src/controllers/contractController.ts#uploadSignedDocument`.
- Multer + new multipart endpoint with a dedicated fileFilter message: when a new endpoint needs a different error message than the existing property-upload JPEG/PNG wording, DO NOT change the existing branch. Throw a new sentinel Error (e.g., `Error('INVALID_PDF_FILE_TYPE')`) in the fileFilter and add a dedicated errorHandler branch that emits the same public code (`INVALID_FILE_TYPE`) with the right message. Public code contract stays stable across endpoints; the message specializes per context. See `src/middlewares/contractDocumentUploadMiddleware.ts` + the mirror branch in `errorHandler.ts`.
- Storage-first + DB-second compensation for single-row updates with filesystem side effects (US-016): for a `prisma.<model>.update` paired with a disk write, write the file FIRST (produces the URL), then the DB. If the DB write throws, the outer catch runs `removeContractDocumentAbsolute(saved.absolutePath)` to clean up the orphan file. Opposite order (DB first, storage second) leaves the DB pointing at a non-existent file if the storage write fails — worse end state. Contrast with prisma.$transaction-wrapped updates: those can save the file inside the tx callback and rely on automatic rollback + explicit storage cleanup in the caller's catch (see `updateProperty` in US-007).
- Env-var-with-default parsing pattern: `const envParsed = envRaw ? Number(envRaw) : NaN; export const LIMIT = Number.isFinite(envParsed) && envParsed > 0 ? envParsed : DEFAULT;` — handles unset, empty, non-numeric, negative, and zero in one expression. Avoid `parseInt(envRaw) || default` because parseInt's leading-digit coercion silently accepts `'15abc'` as 15. Use `Number()` + `Number.isFinite()` for a clean reject-on-noise failure. Applied to CONTRACT_PDF_MAX_BYTES in `src/middlewares/contractDocumentUploadMiddleware.ts`.
- Email/notification transports as dependency-injection seams, not hard deps: when a service is specified as "feature-flagged email stub now, real integration later" (US-017), don't import nodemailer/sendgrid/resend. Export an interface (`SupportEmailTransport { send(envelope): Promise<void> }`) and accept the transport as an optional second argument on each send function. Unit tests inject `{ send: vi.fn() }` with no network. Production can later inject a real client wrapper WITHOUT changing the service surface. The env gate (`SUPPORT_EMAIL_ENABLED`) is separate from whether a transport is wired — the service handles (disabled, any-transport), (enabled, transport=undefined→TODO log), (enabled, transport→call once). Swallow transport failures inside the service (`try/catch` → `logger.error`) since the product contract is "email side-effects are non-fatal to the API request".
- Test-level env state restoration: when a service reads `process.env.X` at call time (not at import time), tests can mutate `process.env.X` inside a test and must restore it in `afterEach` to avoid leaking state to sibling test files. Capture `const ORIGINAL = process.env.X` at module load, then `afterEach(() => { if (ORIGINAL === undefined) delete process.env.X; else process.env.X = ORIGINAL; })`. `beforeEach(vi.clearAllMocks)` is orthogonal and still needed. Applied in `tests/supportEmailService.test.ts` for SUPPORT_EMAIL_ENABLED and SUPPORT_EMAIL_TO.
- Enum pattern for actor-context captures (US-017): when a ticket/log/audit model records a user's role at the time of an event, create a dedicated enum (`SupportUserRole` mirroring the values of the main `Role` enum) rather than reusing `Role` directly. Two reasons: (a) future role semantic changes (split LANDLORD→LANDLORD_VERIFIED/LANDLORD_PENDING) should NOT retroactively change historical context values; (b) Prisma enum deprecations are easier when the enum is not shared across many relations. The cost is one more enum — the win is a clean historical audit trail and safer enum evolution.
- SupportTicket onDelete semantics mirror the schema's established conventions: `userId` (opener) uses `onDelete: Cascade` — deleting a user takes their support tickets with them (same as `Favorite`, `Notification`). `assignedToId` (admin actor) uses `onDelete: SetNull` — deleting an admin preserves their assigned tickets with a null assignee (same as `RentalPayment.updatedBy`). This keeps "opener is the subject" vs. "actor is a pointer" semantically distinct across the schema. Reuse these conventions for any future audit/actor column — no new FK policy required.
- Testing `requireRole('X')` in HTTP integration tests: DO NOT stub `requireRole` to a pass-through. Use `vi.mock('../src/middlewares/authMiddleware', async (importOriginal) => { const actual = await importOriginal(); return {...actual, checkJwt: stub, authSyncMiddleware: stub} })` — spread `actual` so the REAL `requireRole` function is preserved. That way the 401 test case exercises the stubbed `checkJwt` branch, and the 403 test case exercises the REAL `requireRole` operating against a stubbed `req.localUser` with role ≠ the required role. If `requireRole` were stubbed to always `next()`, the 403 test would assert nothing. Reference: `tests/adminSupportTicketsList.test.ts` (US-019).
- Query-param coercion via `z.coerce.number()`: Express delivers `req.query` entries as strings (or arrays of strings for `?a=1&a=2`). `z.coerce.number().int().min(1).default(50)` is the one-liner that handles the common case — empty → default, numeric string → int, array → rejected with 400, non-numeric → rejected with 400. Pair with a custom `refine` at the object level for cross-field rules like `from <= to`. Reference: `src/utils/supportTicketValidation.ts#listSupportTicketsQuerySchema` (US-019).
- `z.string().datetime()` is too strict for calendar-picker inputs: it requires a full ISO 8601 timestamp (`2026-05-07T12:00:00Z`) and rejects the plain `2026-05-07` that HTML `<input type="date">` emits. For API filters, use a custom validator: `z.string().refine(v => !Number.isNaN(Date.parse(v)))` accepts both forms. Then convert to `Date` in the controller via `new Date(v)`, which handles both — `2026-05-07` → midnight UTC, ISO timestamps → exact instant. Reference: `src/utils/supportTicketValidation.ts#isoDateLike` (US-019).
- Paginated list envelope convention for this codebase (US-019): `{ data: T[], page, pageSize, total }` — flat keys, no `meta` wrapper. Contrast with `propertyService.listForModeration` which returns `{ data, meta: { total, page, limit, totalPages } }` — older shape. New paginated endpoints should use the flat shape (matches what `/admin/support/tickets` returns and what the frontend admin panel expects). Client computes totalPages = Math.ceil(total/pageSize) if needed — server doesn't need to precompute it.
- Admin-only listing endpoints — where to register the route: if the resource already has its own route file (e.g., `supportRoutes.ts` already owned POST /support/tickets), ADD the `/admin/*` route there with an inline `const adminOnly = requireRole('ADMIN')` gate — keeps all HTTP surface for the resource in one file. Only use `adminRoutes.ts` when the route doesn't have a natural resource file (metrics, broadcasts, cross-cutting admin tooling). The mount at `app.ts` already applies the global authStack, so inline `requireRole` just adds the 403 gate on top. Reference: `src/routes/supportRoutes.ts#/admin/support/tickets` (US-019).
- Prisma `include` with `select` is a single-JOIN solution for admin list endpoints: `findMany({ where, orderBy, skip, take, include: { user: { select: { id, name, email, role } }, assignedTo: { select: { id, name } } } })` stays 2 queries total (count + findMany), regardless of pageSize. Never replace with `findMany + Promise.all(rows.map(r => prisma.user.findUnique...))` — that's the N+1 anti-pattern the `include + select` pattern exists to avoid. Pair with a `count({ where })` using the SAME where so the envelope's `total` reflects the filters.
- FK existence pre-checks beat P2003 surfacing (US-020): when a PUT/POST accepts an optional `*Id` that references another table, do a `prisma.user.findUnique({ where:{id}, select:{id:true} })` BEFORE the main write and throw a domain error (e.g. `SupportTicketError(400, 'ASSIGNEE_NOT_FOUND', ...)`) on null. Relying on Prisma's P2003 FK violation path instead leaks as 500 INTERNAL_SERVER_ERROR through the global errorHandler — worse end-user surface. Skip the pre-check when the incoming value equals the existing value on the row (no-op short-circuit) so the PUT stays "pay-for-what-you-change". Applied in `supportTicketService.updateForAdmin` — reuse for any future endpoint that accepts a nullable FK in its body.
- Cross-field Zod refines for "at least one field required" + "if X then Y": use `.refine` on the object schema (not on individual fields) to implement both rules declaratively in one schema. `z.object({...all optional}).refine(v => Object.values(v).some(x => x !== undefined), {message}).refine(v => v.status === 'RESOLVED' ? typeof v.resolution === 'string' : true, {path:['resolution'], message})`. Multiple refines stack — each emits its own VALIDATION_ERROR issue if violated, and they share the same fail-fast 400 envelope via the global errorHandler. Reference: `updateSupportTicketSchema` in `src/utils/supportTicketValidation.ts` (US-020).
- Shared view projection helper for GET+PUT consistency (US-020): when an admin endpoint has BOTH a list GET and a single-resource PUT, extract a `toAdminView(row)` helper from the service module and have BOTH paths use it. Benefits: (a) the wire shape for an item is byte-identical between the two endpoints (frontend caches can be keyed by id without checking "is this from the list or the detail?"), (b) ISO date serialization / nullable field handling / nested `user`+`assignedTo` projection is owned by ONE function, (c) adding a new field to the view updates both endpoints with zero drift. Avoid per-endpoint inline `.map(r => ({...}))` once a second consumer appears.
- Synthesize minimal shape for downstream non-fatal side-effects (US-020): when a controller has a `SupportTicketAdminView` but the downstream email service signature expects a full `SupportTicket`, construct a minimal object at call site with the fields the email service ACTUALLY reads (checked by grep of the service module) — don't make a second DB round-trip for "email compatibility". The `as any` cast on the userRole field is acceptable here because the string values flow through unchanged (SupportUserRole enum values ARE the strings) and the email service treats the field as `${ticket.userRole}` interpolation anyway. Saves ~5ms of latency on the admin PUT hot path without hurting any consumer.

- **Learnings for future iterations:**
  - FK existence check beats P2003 surfacing. When a write accepts `assignedToId` (or any optional FK pointer), read the target table first with a tiny `{select:{id:true}}` and throw a domain 400 error. The default Prisma P2003 path goes through errorHandler as 500 INTERNAL_SERVER_ERROR, which is the wrong status. Skip the pre-read when the incoming value equals the existing column value — no-change writes shouldn't cost a second query.
  - Cross-field Zod refines stack. Chain multiple `.refine(…)` calls on the same object schema instead of writing one giant composite refine function. Each emits its own issue in the ZodError.errors array, the global errorHandler flattens them all into the 400 envelope, and future readers can delete one rule without untangling the others.
  - Shared view projection across endpoints. When a resource has both a LIST and a DETAIL endpoint, extract `toAdminView(row)` to the service module and have both paths call it. Byte-identical responses between GET and PUT make frontend cache management trivial — the PUT response can overwrite the list item in place without re-fetching.
  - Synthesize downstream shapes at the call site. When a controller needs to call a side-effect (email, audit, notification) that expects a full DB row, don't re-fetch — construct a minimal object from the view that contains ONLY the fields the side-effect actually reads. Saves a DB round-trip on the hot path; typecheck catches any missing field via the SupportTicket type signature.
  - Path-param validation separate from body validation. `z.string().uuid().safeParse(req.params.id)` BEFORE `req.body.parse` keeps error paths semantic (`id` vs `status`/`resolution`/`assignedToId`). Without the safeParse the Zod chain collapses a non-UUID id into a 400 with an empty path, which the frontend can't key UI errors off.
