- **Learnings for future iterations:**
  - PG conf paths on Debian/Ubuntu live at `/etc/postgresql/${PG_MAJOR}/main/{postgresql,pg_hba}.conf` regardless of where the data dir is. Hard-coding `${PG_CONF_DIR}=/etc/postgresql/${PG_MAJOR}/main` is the standard expectation for `apt`-installed clusters and what every operator runbook on this stack assumes — don't try to discover it dynamically with `pg_lsclusters`.
  - `CREATE DATABASE` does NOT accept `IF NOT EXISTS`. Idempotency requires a pre-flight `SELECT 1 FROM pg_database WHERE datname=…` check. Same for `CREATE ROLE` — guard via `DO $$ ... IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname=…) THEN CREATE ROLE … END IF; END $$;` to survive re-runs.
  - For postgresql.conf/pg_hba.conf .bak preservation across re-runs, do NOT use `sed -i.bak` (it overwrites .bak with the already-modified file on rerun). Pattern used: `cp -p` to .bak only when .bak does not yet exist, then sed without the suffix (or with a scratch `.tmp` that is immediately deleted). Future scripts (04-cutover-env.sh) should follow the same idiom for `.env.pre-supabase-cutover.bak`.
  - The PGDG repo setup uses the keyring at `/etc/apt/keyrings/postgresql.gpg` and a `signed-by=` source entry — this is the modern (post-`apt-key`) pattern. Don't regress to `apt-key add`.
  - Smoke-test caveat: when the role pre-existed (re-run with no fresh password generated), the script cannot test `\conninfo` over TCP without `PGPASSWORD` from the operator. The script warns rather than fails — re-runs after the first successful run should not be blocked by a missing password.
