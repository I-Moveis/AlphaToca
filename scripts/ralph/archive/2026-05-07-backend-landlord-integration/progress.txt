## Codebase Patterns

The following patterns were consolidated during the US-001‚ÜíUS-020 epic and remain load-bearing for LL-001‚ÜíLL-022. New LL-specific patterns should be appended here after each iteration.

- Prisma enum names are globally unique in a schema ‚Äî you cannot define two enums with the same name. `PaymentStatus` is already taken by `TenantPayment` (PENDING|PAID|OVERDUE|CANCELLED); the monthly rent status enum lives under `RentalPaymentStatus` (AWAITING|PAID|LATE) on `RentalPayment`. If a PRD asks for an enum name that's already bound, pick a namespaced alternative (`<Model>Status`) and document the deviation ‚Äî don't rename the incumbent.
- Hand-authored migration verification: before committing a hand-authored `prisma/migrations/<stamp>_<name>/migration.sql`, run `npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script` and grep for your table name. The generated DDL for the new table/index/FK should match the hand-authored file byte-for-byte (ENUM CREATE order, VARCHAR(n) vs TEXT, DEFAULT 'X', index/constraint names, CASCADE vs SET NULL on FK). `prisma migrate diff` is read-only and safe to run against any URL ‚Äî including the live Supabase pooler ‚Äî since it operates on the schema file, not the DB.
- `.env` DATABASE_URL points at the live Supabase project ‚Äî DO NOT run `prisma migrate dev/deploy/reset` locally; hand-author `prisma/migrations/<stamp>_<name>/migration.sql` instead and verify with `npx prisma validate` + `npx prisma generate`. Reviewer runs the actual `migrate reset --force` on a disposable dev DB before landing.
- Relation onDelete conventions in this schema: child-of-Property tables use `onDelete: Cascade` (property_images, favorites, property_expenses, maintenance_logs, notifications, rental_payments) so a property delete takes the subgraph with it. User-is-actor FKs (e.g., `RentalPayment.updatedBy`) use `onDelete: SET NULL` so deleting a user preserves historical rows with a null actor. Avoid `Restrict` unless the business rule actively forbids deletion (contracts use Restrict because a pending contract should block landlord/property/tenant deletion).
- Seed IDs: random seeds use `crypto.randomUUID()` (not `faker.string.uuid()`); stable demo IDs live in `prisma/demoIds.ts` as canonical UUIDv4 constants ‚Äî don't replace those constants with `randomUUID()` calls because they're referenced by tests and Firebase UID mappings.
- UUID validation: routes use `z.string().uuid()`; any seed id you introduce must match `/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i`.
- Test runner: `npx vitest run` (vitest config in `vitest.config.ts`). Many pre-existing tests fail due to an unfinished Auth0‚ÜíFirebase migration (`userService.upsertUserFromAuth0` missing, `tests/auth.test.ts`, `tests/ingestKnowledge.test.ts`) ‚Äî treat those as baseline noise, not regressions.
- Grep scope for "seed" lookups: include `prisma/` (generators.ts, demoData.ts, demoIds.ts, seed.ts) ‚Äî seeds are not under `src/` in this repo.
- Typecheck: `npx tsc --noEmit` ‚Äî must pass before commit.
- Swagger is split: `src/config/swagger.ts` owns component schemas (e.g., `Property`); per-route JSDoc blocks in `src/routes/*.ts` own request/response shapes ‚Äî when an enum changes, both files need an edit.
- Prisma enum value rename/add: use `ALTER TYPE "X" ADD VALUE 'NEW';` (for additions, used by LL-019) or `ALTER TYPE "X" RENAME VALUE 'OLD' TO 'NEW';` (for renames) ‚Äî both are metadata-only and rewrite the catalog label in place, zero data loss, zero downtime.
- `propertyValidation.ts` uses `z.nativeEnum(PropertyStatus)` ‚Äî it re-reads the Prisma client enum at import time, so bumping the enum only requires a `prisma generate`, not a Zod edit. Same convention for `rentalPaymentValidation.ts` ‚Üí `RentalPaymentStatus` (US-010). Applies to LL-019 (PropertyType), LL-016 (ContractDocumentStatus), LL-018 (VisitSource).
- Integration tests that need a "backing store" without a real DB: mount the real Express app via `supertest` and swap `propertyService` with `vi.mock('../src/services/propertyService', importOriginal)` spreading the actual module plus overriding only the methods you exercise. A `Map<id, record>` in `beforeEach` is enough to simulate create/update/get/search chains ‚Äî Prisma model semantics are preserved by having the mock merge updates (`{...existing, ...data}`).
- Route-level auth: `app.use('/api', propertyRoutes)` in src/app.ts mounts WITHOUT the `authStack`, so POST/PUT/DELETE /properties are currently unauthenticated at the Express layer. Owner-only checks will need the middleware added OR an inline `localUser`/ownership guard in the controller.
- Zod error shape: the global errorHandler (`src/middlewares/errorHandler.ts`) already converts `ZodError` into `{ status: 400, code: 'VALIDATION_ERROR', messages: [...] }` ‚Äî route/controllers only need to `schema.parse(req.body)` and pass the ZodError to `next(error)`.
- `ProcessStatus` on `RentalProcess` in this schema is `TRIAGE|VISIT_SCHEDULED|CONTRACT_ANALYSIS|CLOSED` ‚Äî there is NO `ACTIVE` value. When a PRD/handoff says "RentalProcess with status=ACTIVE", the semantic the codebase actually implements is the `Contract` model (ContractStatus=ACTIVE|TERMINATED|COMPLETED). `contractService.createContract` already couples an ACTIVE Contract insert with `Property.status='RENTED'` in one transaction ‚Äî so Contract is the source of truth for "who is the current tenant".
- To add a relation-derived field (e.g., currentTenant) to a property read path without N+1: use `include: { contracts: { where: { status: 'ACTIVE' }, select: { tenant: { select: { id, name } } }, take: 1 } }` in Prisma findUnique/findMany (single JOIN), and for any raw `$queryRaw` path in the same service, emulate the include with ONE batched `prisma.contract.findMany({ where: { status: 'ACTIVE', propertyId: { in: ids } } })` followed by a Map join in memory ‚Äî still constant queries per page.
- Mocking Prisma for service-level tests: `vi.mock('../src/config/db', () => ({ default: { model: { method: vi.fn() }, $queryRaw: vi.fn() } }))` at the TOP of the test file (before any import that pulls the service in), then cast with `(prisma.model.method as any).mockResolvedValue(...)` per test. See `tests/propertyCurrentTenant.test.ts` for the full template ‚Äî works cleanly with `src/config/db`'s `export default prisma` shape.
- Mocking `prisma.$transaction(async (tx) => ...)`: the factory returns `{ default: { ..., $transaction: vi.fn(async (fn) => fn({ model: prismaMocks })) } }`. Default impl proxies the `tx` object to the same top-level mocks so callees can use either `tx.model.method` or `prisma.model.method` interchangeably in tests.
- Content-Type-conditional multer: to let a single route accept BOTH application/json AND multipart/form-data, wrap `multer.array(...)` in a small handler that only invokes multer when `req.headers['content-type']` starts with `multipart/form-data` (case-insensitive), otherwise `next()`. `src/middlewares/uploadMiddleware.ts#conditionalPropertyPhotoUploadHandler` is the reference ‚Äî reuse for LL-022 multipart amenity support.
- Auth gating on the property route group: `app.use('/api', propertyRoutes)` deliberately mounts WITHOUT the global authStack ‚Äî many GET paths (search, /:id, list) are public. To gate a SPECIFIC route on that router, declare `const authStack = [checkJwt, authSyncMiddleware]` INSIDE `src/routes/propertyRoutes.ts` and spread it per-route (e.g., `router.put('/properties/:id', ...authStack, uploadHandler, controller.update)`). Ownership is enforced INSIDE the controller via `req.localUser.id === property.landlordId` ‚Äî this avoids the 404-vs-403 leak (404 first if the property doesn't exist, then 403 if it does but you're not the owner). This is the pattern for LL-005, LL-008, LL-009, LL-016-PATCH.
- New resource ‚âÝ new mount pattern: when a story adds a resource that's unrelated to any existing route file (e.g., LL-002 `/landlord/metrics`, LL-011 `/conversations` list), prefer a NEW `src/routes/<resource>Routes.ts` mounted at `app.ts` via `app.use('/api', authStack, <resource>Routes)` ‚Äî not a local `authStack` spread inside the route. The local-authStack pattern (propertyRoutes.ts) exists ONLY because that router has mixed public+private paths. New resources with uniform auth go under the global `authStack` mount for less boilerplate. See `src/routes/conversationRoutes.ts` for the minimal template.
- Query-param validation via Zod: use `schema.parse(req.query)` inside the controller. Express serializes query strings as plain `Record<string, string | string[] | ...>`, so `z.string().uuid()` works out of the box. Repeated params (`?a=1&a=2`) arrive as arrays; if a param should be single-valued, either enforce `z.string()` (which will reject arrays with a ZodError ‚Üí 400) or preprocess to take the first element. Missing params arrive as `undefined` ‚Äî rejected by required Zod fields with a readable path.
- Server-derived identity for upsert keys: when a write path has a compound unique key and one of the components is derivable from server state (e.g., `landlordId` from `Property.landlordId`), NEVER accept it from the request. Derive it in the controller from a prior read (`propertyService.getPropertyById(propertyId)`), then pass it to the service.
- "Create-or-get atomic" contract via `prisma.<model>.upsert` with `update: {}`: when the business semantics are "return the existing row unchanged if present, otherwise create it", an upsert with an empty `update:` object is the right tool. The DB unique constraint converts the duplicate-INSERT race into a SELECT transparently.
- Period-computation convention (US-009+): the server-side `YYYY-MM` of the current month is computed in UTC via `new Date().getUTCFullYear()` + `getUTCMonth()+1` zero-padded. Pattern is centralized as `currentPeriod(now?: Date): string` in `src/services/rentalPaymentService.ts` ‚Äî LL-003/LL-004/LL-005/LL-009 should all consume it for month boundaries instead of reimplementing.
- Prisma `upsert` on a compound-unique key: the `where` clause reuses the SAME client key used by `findUnique` (the `@@unique([...], name: "...")` value, e.g., `rental_payments_property_period_key`).
- Model-extend vs model-create decision (US-013): before creating a new Prisma model, grep for every existing model that shares (a) the same relation targets, (b) the same status/lifecycle field, and (c) the same invariants enforced by services. If an incumbent model checks all three, extend it with the missing columns ‚Äî never create a parallel. For LL-010 (ConversationMessage) the decision is explicit: the incumbent `Message` model is tied to `ChatSession` (WhatsApp RAG) and fusing would force nullable FKs ‚Äî so a NEW model is the right call. Document the rationale in the migration SQL header.
- WebSocket service pattern: `src/services/chatSocketService.ts` is the canonical template for socket emitters. It wraps `getIO().to(room).emit(event, data)` in a `safeEmit` that logs but never throws ‚Äî use the same shape for LL-014's `conversationSocketService`. Rooms `user:${id}` and `landlord:${id}` are joined on connection in `src/config/socket.ts`; no registration code needed in the emitter.
- Post-commit storage cleanup is fire-and-forget: after a Prisma `$transaction` that modifies rows linked to files on disk, run `await cleanup(...).catch(err => logger.warn(...))` OUTSIDE the tx so a storage failure never rolls back committed DB state.
- `PropertyError` lives in `src/services/propertyService.ts` and mirrors `ContractError`/`VisitError`/`ProposalError` ‚Äî `httpStatus, code, message`. The controller pattern is `if (err instanceof PropertyError) res.status(err.httpStatus).json({ status, code, messages: [{ message }] })` right inside the try/catch, BEFORE `next(err)`. Reuse this class for any future 4xx business rule in property service. Don't add a new error class for each story.
