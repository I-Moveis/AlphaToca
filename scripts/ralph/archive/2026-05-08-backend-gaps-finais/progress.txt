- **Conversation schema uses landlordId/tenantId columns, not a `participants[]` array**. When an AC asks to "exclude req.localUser.id from participants", translate that to `WHERE landlordId = userId OR tenantId = userId` + `counterpart = (row.landlordId === userId ? row.tenant : row.landlord)`. The schema is intentionally denormalized this way (single UNIQUE index on `(propertyId, landlordId, tenantId)` anchors the "one 1:1 thread per property per pair" invariant). Do NOT add a participants array — it would duplicate state and break the upsert contract in `conversationService.resolve`.
- **Existence-hiding 404 is the project-wide convention for conversation endpoints, not 403**. LL-012, LL-013, and LL-015 (`GET/POST /conversations/:id/messages`, `POST /conversations/:id/read`) all return 404 for both "conversation doesn't exist" AND "caller is not a participant" — never 403. This is a deliberate security posture: a 403 vs 404 oracle lets an authenticated attacker enumerate valid conversation UUIDs by probing. When a P4 AC says "403 for non-participant" for a conversation endpoint, prefer the existence-hiding 404 that's already locked by tests and JSDoc — don't downgrade the guard to match AC wording. Update the PRD notes to document the deviation.
- **Chat pagination returns ASC-within-a-page even though DB orderBy is DESC**. LL-012's `conversationService.listMessages` queries `orderBy: { createdAt: 'desc' }` + `take: limit` (efficient for "latest N" / "N before cursor"), then does `rows.reverse()` in-memory to hand the client ASC within the page. The cursor pagination DIRECTION (older pages via `before`) is still DESC-of-pages. When an AC says "ordered createdAt DESC", translate to "paginated DESC-of-pages but items within a page are ASC" — that's the contract the frontend chat UI expects (oldest-at-top, newest-at-bottom). Tests in `tests/conversationMessagesService.test.ts` lock this in.
- **Cursor-forging guard in listMessages**: when `before` is provided, `findUnique({ where: { id: before } })` is called and the result's `conversationId` is compared against the path's `:id`. If they differ (or the cursor doesn't exist), the service returns `{ messages: [], markedReadIds: [] }` — "end of list" semantics. This prevents a caller from passing a message id from ANOTHER conversation as the `before` cursor to probe timestamps. When adding new cursor-based endpoints, replicate this guard.
- **`prisma migrate diff` without a shadow DB**: this repo's `.env` DATABASE_URL points at Supabase remote (no local shadow available). The Codebase Patterns line "verify byte-for-byte with `--from-empty --to-schema-datamodel`" gives the full schema DDL, which is noisy when you only want the new-change diff. Instead: `git show HEAD:prisma/schema.prisma > /tmp/schema-before.prisma && npx prisma migrate diff --from-schema-datamodel /tmp/schema-before.prisma --to-schema-datamodel prisma/schema.prisma --script`. That emits ONLY the ALTER/CREATE statements for the current uncommitted edits — which is exactly what you want to paste into the hand-authored `migration.sql` before adding any backfill UPDATEs.
- **$transaction mock pattern for service tests**: when a service uses `prisma.$transaction(async (tx) => {...})`, mock prisma so that `$transaction` invokes the callback with a `tx` object that PROXIES the SAME mocks the test asserts against. Example: `const messageCreate = vi.fn(); const $transaction = vi.fn(async (fn) => fn({ conversationMessage: { create: messageCreate } }))`. This lets a single `expect(messageCreate).toHaveBeenCalled(...)` assertion catch calls made inside the tx callback, without the test needing to distinguish tx-vs-non-tx prisma handles. Pattern lives in `tests/propertyPhotosRemove.test.ts` (used for $transaction on property photos) and now `tests/conversationMessageCreateService.test.ts` (createMessage + Conversation.lastMessageAt update).
- **Conversation.lastMessageAt is now the authoritative ordering timestamp** (US-006). Nullable column populated by `conversationService.createMessage` inside a `$transaction` with the INSERT. `conversationService.list` prefers `row.lastMessageAt ?? lastMsg?.createdAt ?? row.createdAt`. When adding new chat/thread endpoints that need "last activity" ordering, read the column directly; do NOT re-introduce an `include: { messages: { take: 1 } }` subquery. Empty threads keep `lastMessageAt = null` — that's intentional (distinct from "has one ancient message"); the list fallback handles the sort correctly via `createdAt`.
- **Injectable `now: Date = new Date()` for any time-branching service** (US-007 `rentalPaymentService.listByTenant`, also used by `getCurrent`/`upsertCurrent`). Real callers never pass the third arg — tests pin it to exercise deterministic "past vs current month" branches. Apply this pattern to US-010 metrics (30-day window) and US-011 analytics (30d/90d/1y windows) — do NOT call `new Date()` inline in a service that gets tested.
- **Month comparison via lexicographic string compare**: `period < currentPeriod` is the correct and cheapest way to check "is this month in the past". `YYYY-MM` strings sort chronologically because of the fixed-width format. No Date parsing needed in hot paths. Same trick applies to `YYYY-MM-DD` — avoid `Date.parse` unless you actually need arithmetic.
- **Synthesis pattern for "one row per month" histories**: when an AC asks for a dense monthly series (payments history, analytics dailies), build the period list FIRST from a source-of-truth (contract window, query window), then LEFT-JOIN-style merge DB rows on top. Missing periods get synthesized with deterministic defaults. The reverse (start from DB rows and fill gaps) is always more code and more bugs. Pattern lives in `rentalPaymentService.listByTenant` (US-007) — copy for US-010/US-011.
- **Role gating goes at the MOUNT, not inside routers**: `src/app.ts:142` mounts landlord-only routes with `app.use('/api', authStack, requireRole(Role.LANDLORD), landlordRoutes)`. The router file itself (`src/routes/landlordRoutes.ts`) does not re-apply auth or role middleware — all routes inside are automatically LANDLORD-only. When adding a new landlord-only endpoint, put the route handler inside this router; do NOT gate it inline in the handler.
- **Prisma P2021 graceful fallback**: when a counter/query targets a model whose table might not exist yet (mid-rollout DB drift), wrap the Prisma call in a try/catch on `Prisma.PrismaClientKnownRequestError` + `err.code === 'P2021'` and return a neutral value (0 for counts, [] for lists). Pattern lives in `landlordController.countUnreadMessages` (LL-002) — useful when an endpoint launches in a commit series before its dependency migration ships. Once the table lands, the catch becomes dead code but is harmless to leave as belt-and-braces; don't rip it out in a no-op closure.

When US-013 runs, it must NOT add a duplicate model — it needs to either (a) extend the existing model with `category` + rename `sentAt` → `receivedAt` via a hand-authored migration, or (b) add the missing fields while keeping `sentAt` as an alias. Check `grep -n NotificationType prisma/schema.prisma` + look at all existing callers before writing. Also: the Notification model uses `@@index([userId, readAt])`, whereas the AC asks for `@@index([userId, receivedAt])` — the index needs updating or adding. Do NOT ignore the existing model.
- **Telemetry tables for property analytics already exist** (PropertyViewEvent at prisma/schema.prisma:658, ContactClickEvent at prisma/schema.prisma:676). Both are fed by public-facing endpoints: `POST /properties/:id/view` (propertyController.recordView) and `POST /properties/:id/contact-click` (propertyController.recordContactClick). When a future AC says "telemetry doesn't exist yet — return 0/[] with TODO", first `grep -rn 'propertyViewEvent\\|contactClickEvent' src/ prisma/` — if the instrumentation landed in a prior rollout (LL-008 did it), the real counts are already wired and returning zeros would be a regression. The existing `propertyAnalyticsService.getAnalytics` does 6 counts + 1 raw GROUP BY day in `Promise.all` with zero-filled `dailyViews` via `enumerateDays(fromInclusive, toExclusive)`.
- **Zero-filled daily bucket pattern**: `enumerateDays(fromInclusive, toExclusive)` + `dailyMap.get(date) ?? 0` is the correct pattern for dense daily series (analytics charts, heatmaps). The DB GROUP BY returns only days with events; the enumerate-and-merge step fills the gaps so the client can `.map((b, i) => ...)` without bucket lookups. Same pattern as US-007's monthly synthesis for rent payments — it's the "synthesis pattern for 'one row per time-bucket' histories" generalized to days. `startOfUtcDay` + `addDaysUtc` + `formatYyyyMmDd` are the companion UTC helpers.
- **UTC-exclusive end-of-range**: analytics services use `toExclusive = addDaysUtc(startOfUtcDay(now), 1)` and `fromInclusive = addDaysUtc(toExclusive, -WINDOW_DAYS[window])` for the window bounds, then pass `{ gte: fromInclusive, lt: toExclusive }` to Prisma. The exclusive end-of-today ensures events recorded during the current day still land in today's bucket (avoiding the "truncated at start-of-day(now)" off-by-one). Apply this to any future time-window count/aggregate endpoint.
- **Scope-gated field downgrades belong in the CONTROLLER, not the service** (US-012). When an AC says "if req.localUser lacks the X scope, force field=SAFE_VALUE", the check lives next to where `req.localUser` is accessible — the controller layer. The service stays a pure persistence layer that honors whatever it's handed, so the future scoped flow (when the scope actually lands) persists the real value without re-plumbing the service signature. Pattern lives in `visitController.create` via `callerHasAiAgentScope(req)` + `effectiveSource = hasScope ? input.source : VisitSource.MANUAL`. When the scope exists, flip the helper from `return false` to a real claim check — service layer and all service-level tests stay untouched.
- **Test layer mirrors guard layer** (US-012). Service-level tests (`tests/visitSource.test.ts`) assert the pure-persistence contract: "if called with source=AI, persist AI" — that's what the service does and will always do. Controller-level tests (`tests/visitController.test.ts`) assert the end-to-end HTTP contract: "if the body sends source=AI from a user without the scope, the service is called with source=MANUAL". Use `not.toHaveBeenCalledWith(expect.objectContaining({ source: 'AI' }))` as a sanity-check assertion to lock in that the bad value never reached the service. When updating a test's describe block to clarify layer semantics, also add a docblock inline explaining the layer boundary so the next reader doesn't restore the "fix" by aligning the service test with the controller test.
- **Postgres `CREATE TYPE ... AS ENUM` is strictly stronger than a `CHECK` constraint**, and it's the canonical Prisma way. When an AC asks for a CHECK constraint matching an enum's values, the enum column is sufficient — the DB rejects non-enum values at the type level (before any row is inserted), with compile-time safety in the generated Prisma client as a bonus. Do not add a separate `ALTER TABLE ADD CONSTRAINT CHECK (col IN (...))` — it's redundant with the enum type and creates a maintenance burden (CHECK + enum must be kept in sync manually).
- **Data-preserving field renames use `ALTER TABLE RENAME COLUMN`, NOT DROP + ADD** (US-013). `npx prisma migrate diff` emits `DROP COLUMN "x", ADD COLUMN "y"` whenever a field is renamed in `schema.prisma` — it can't tell the two are related, so it assumes the old data should be thrown away. On a live DB this would delete every row's history. The hand-authored migration must translate that back to `ALTER TABLE "t" RENAME COLUMN "x" TO "y";`. Lock the intent with a migration-SQL assertion test that explicitly asserts `.not.toMatch(/DROP COLUMN "sent_at"/)` so a future Prisma regeneration can't silently revert to the destructive pair. Pattern lives in `tests/notificationMigration.test.ts`.
- **Cross-device history persists for ALL users, not just users with an FCM token** (US-013 broadcast). A "cross-device notification history" is only cross-device if it survives the FCM push — i.e., if a user opens the app tomorrow on a device that was offline today, they should still see the broadcast. `broadcastService.sendToAll` now SELECTs every user via `prisma.user.findMany({ select: { id, fcmToken } })` and creates one Notification row per user via `createMany` inside a `$transaction`, THEN dispatches FCM to only the token-holders. If persistence throws (DB down), the FCM dispatch is NEVER attempted — the DB is the source of truth, and a push without a history row is worse than no push at all (the user clicks the push, app opens to /notifications, list is empty → confusion).
- **Bare array responses for per-user list endpoints** (US-003 support tickets, US-013 notifications). The project convention is: GET endpoints that return a user's own list (support tickets, notifications) return a bare JSON array; admin/paginated endpoints return `{ data, total, page, ... }` envelopes. The /notifications endpoint's OLD shape was `{ data: [...] }` with the full Prisma row; US-013 switched it to a bare array of `NotificationView` (narrow projection) — same move US-003 made. When an AC specifies a bare array shape, do NOT wrap it in an envelope even if the pre-existing endpoint did — the AC shape IS the contract.
- **`NotificationType` vs `NotificationCategory` are intentionally different enums** (US-013). `NotificationType` is the FCM-dispatch taxonomy (VISIT_SCHEDULED, DOCUMENT_REJECTED, BROADCAST, etc. — one per business trigger); `NotificationCategory` is the coarser UX bucket the /notifications screen uses to colour-code groups (update / announcement / system). A future refactor might be tempted to merge them — don't: the FCM dispatch code needs fine-grained types for deep-link routing, while the UX needs a 3-way bucket for grouping. Keep them separate; the existing `type` column stays on the Notification model as a secondary field for the future-case where the /notifications screen needs to deep-link into the trigger.
- **Additive verb expansion instead of contract replacement** (US-014). When an AC asks for a new HTTP verb on an existing path (`PUT /notifications/:id/read` next to the pre-existing `PATCH /notifications/:id/read`), ADD the new endpoint alongside the old one rather than mutating the old endpoint's response shape / status code. PATCH returned 200 + `{ data: row }`; PUT returns 204 + empty body with idempotency semantics. Both coexist; Flutter callers migrate at their own pace. Cost: ~8 lines of JSDoc duplication. Benefit: zero breaking change for production builds already in the field. Do NOT aggressively refactor the legacy endpoint during the same story — that's a separate 'deprecate PATCH' ticket.
- **Idempotency on mark-read is a cross-device correctness invariant** (US-014). When the current row's `readAt` is non-null, the handler MUST NOT overwrite it — preserving the first-read timestamp is what makes the /notifications screen's "read 2 days ago" grouping meaningful on a second device. Gate the mutation with `if (row.readAt === null) prisma.update(...)` and lock it in tests with an explicit `expect(mockUpdate).not.toHaveBeenCalled()` on the already-read branch. Pattern lives in `notificationController.markAsReadIdempotent`.
- **Narrow `select:` on ownership-check findUnique** (US-014). When a handler only needs `id`, `userId`, and maybe one status field to decide 404/403/mutate, do not pull the full row — pass `select: { id: true, userId: true, readAt: true }`. Shipping less from the DB on a hot path is free, and it makes the assertion in the accompanying test crisper (the test can lock the exact 3-field select shape as a regression tripwire). Apply to any controller that follows the `find → check ownership → maybe mutate` pattern.

- **Capping enumeration at current month for ACTIVE contracts**: `effectiveEnd = c.endDate < currentMonthStart ? c.endDate : currentMonthStart`. This prevents rendering "phantom" rows for future months of ACTIVE contracts (a landlord with a 2-year contract shouldn't see 24 AWAITING rows today — only the months that have actually elapsed plus the current one). TERMINATED/COMPLETED contracts with endDate in the past pass through unchanged.
- **Mock prisma directly when a service is the SUT**: `tests/propertyPaymentsHistory.test.ts` mocks `src/config/db` to stub `contract.findMany` + `rentalPayment.findMany` so the REAL service runs, exercising the synthesis logic end-to-end. The sibling `rentalPaymentsList.test.ts` takes the opposite approach — mocks the service entirely — because it's testing the controller glue. Pick the boundary based on what's under test.
- **Test data that covers synthesis**: when writing tests for a service that merges DB data with synthesis, make sure the test scenarios include (a) every period has a DB row (pure DB path), (b) no period has a DB row (pure synthesis path), (c) mixed — some present, some missing (the merge path). Test #1 covers (c), test #5/#7 cover (b), and the existing LL-009 test #2 covers (a).
- **Migration-SQL assertion test as a shadowless `prisma migrate diff` lock**: `tests/contractDocumentStatus.test.ts:230-252` reads the migration file from disk and asserts the exact `CREATE TYPE` + `ALTER TABLE ADD COLUMN` + backfill `UPDATE` clauses are present. This is the durable equivalent of running `prisma migrate diff` in CI — it fires every test run, catches accidental edits to the migration body, and doesn't require a shadow DB. When an AC asks to 'verify migration byte-for-byte', consider writing this kind of assertion test rather than (or in addition to) a one-shot local `migrate diff` — the test stays green as a tripwire long after the commit lands.
- **PATCH vs PUT for single-field updates**: the project convention is PATCH (`/contracts/:id/document-status`, `/contracts/:id/status`) for single-field mutations, reserving PUT for resource-replacement semantics (like `/contracts/:id/signed-document` which uploads a whole new PDF). When an AC says 'PUT /api/resource/:id accepts X in body', check whether a sibling PATCH already exists — if so, the PATCH IS the AC, and introducing a parallel PUT would duplicate coverage for zero behavioral gain.
- **Before writing contract seeds for a field**: `prisma/seed.ts` does NOT create any `Contract` records (only Users, Properties, PropertyImages, ChatSessions, Messages, KnowledgeDocuments). `demoData.ts` and `generators.ts` also have zero contract references (verified with `grep -c 'contract\\|monthlyRent'` → 0/0). When an AC asks to 'update existing contract seeds with coherent values', the answer for this codebase is 'there are no contract seeds'. Don't invent them — that's a different PRD.

## 2026-05-07 - US-014 (PUT /api/notifications/:id/read — idempotent cross-device mark-read)

### What was implemented

Added `PUT /api/notifications/:id/read` as the AC-aligned cross-device contract,
next to the pre-existing `PATCH /notifications/:id/read` (kept unchanged so no
current frontend caller breaks). The new endpoint is idempotent: if the row was
already read on another device, the second `PUT` does NOT overwrite `readAt` —
that's the whole point of "cross-device history", the timestamp must reflect the
first read, not the latest sync.

- New controller method `notificationController.markAsReadIdempotent` (src/controllers/notificationController.ts):
  - 401 guard for missing localUser (uses the same envelope as `list`).
  - `prisma.notification.findUnique({ where: { id }, select: { id, userId, readAt } })`
    — narrow select; we don't need `title`/`body`/`category` for an ownership
    check, and shipping less data from the DB on a hot path is free.
  - 404 when row is missing.
  - 403 when `notification.userId !== req.localUser.id` (explicit 403/404 split,
    see deviation note below).
  - Idempotency gate: `if (notification.readAt === null)` → run `prisma.notification.update({
    where: { id }, data: { readAt: new Date() } })`; otherwise skip the write.
  - `res.status(204).send()` on the happy path. Matches the project convention
    (userController / favoriteController / propertyController / visitController
    all use `res.status(204).send()` for "success, no body").

- Route wired in src/routes/notificationRoutes.ts below the existing PATCH,
  inheriting `authStack` via the existing `app.use('/api', authStack, notificationRoutes)`
  mount in src/app.ts. No app.ts change needed.

### Why a separate endpoint instead of modifying the existing PATCH

The existing `PATCH /notifications/:id/read` is the LL-rollout legacy endpoint —
it returns 200 with the full updated row (ENVELOPE shape `{ data: <row> }`) and
overwrites `readAt` unconditionally on every call. Changing its contract now to
match US-014 would be a breaking change for any existing Flutter call site that
either (a) reads `res.body.data.readAt` to render the mark-read timestamp locally,
or (b) checks `res.status === 200` vs `=== 204`. The safer move is to add PUT
as the new cross-device contract (204, idempotent) and let the Flutter chat PR
migrate callers at their own pace. Eventually the PATCH can be deprecated, but
not as a side-effect of this story.

### Intentional AC deviations (documented)

1. **Kept the 403/404 split explicit** — AC says '403 for non-owner, 404 for
   missing id'. This does NOT follow the chat-family existence-hiding convention
   from LL-012/LL-013/LL-015 because notification ids are not user-enumerable —
   there is no `POST /notifications/resolve` creation surface and no seed
   mechanism that would let an authenticated attacker enumerate valid notification
   UUIDs by probing 403 vs 404. Separately, the frontend relies on
   distinguishable 403 vs 404 for UX copy ('você perdeu acesso a esta
   notificação' vs 'essa notificação foi apagada'). So aligning with the AC is
   correct here even though chat endpoints take a different posture.

2. **Kept the legacy PATCH endpoint alongside the new PUT.** The AC says 'Add
   PUT', not 'replace PATCH'. Both endpoints exist now. Migration of Flutter
   callers to PUT is a frontend ticket, not a blocker for US-014.

- `src/controllers/notificationController.ts` — added `markAsReadIdempotent` method.
- `src/routes/notificationRoutes.ts` — added `router.put('/notifications/:id/read', ...)` + Swagger block.
- `tests/notificationMarkRead.test.ts` — new test file, 7 cases.
- `scripts/ralph/prd.json` — flipped US-014 passes: true + notes.

### Learnings for future iterations

- **When an AC asks for a new verb on an existing path, prefer adding the new
  verb alongside the old one if the old contract is locked by production
  callers.** `PUT` and `PATCH` on `/notifications/:id/read` now coexist —
  different status codes (204 vs 200), different body shapes (empty vs
  `{ data: row }`), different idempotency semantics. The cost is a few lines
  of JSDoc duplication; the benefit is zero breaking change for any Flutter
  build already in the field.

- **Idempotency on mark-read is cross-device correctness, not just politeness.**
  If the second call overwrote `readAt`, the UI on device B would show the
  read timestamp as "now" instead of "when you first read it on device A" —
  that's a meaningful UX regression for the /notifications screen's grouping
  logic ('read 2 days ago' vs 'read just now'). Lock idempotency with an
  explicit `not.toHaveBeenCalled()` assertion on the update mock for the
  already-read branch.

- **`findUnique` with a narrow `select:` on ownership checks.** When a handler
  only needs `id`, `userId`, and maybe one status field to decide 404/403/mutate,
  do not pull the full row. Shipping less data from the DB on a hot path is
  free optimization with zero behavioral cost. Pattern now lives in
  `notificationController.markAsReadIdempotent`.

- **Pre-flight 401 guard before touching `req.localUser.id`.** The existing
  `markAsRead` method reads `(req as any).localUser.id` unconditionally — fine
  when the route is ALWAYS behind authStack, but if authStack is ever swapped
  for a route-level middleware chain that doesn't guarantee `localUser`, the
  crash is a 500 instead of a clean 401. The new `markAsReadIdempotent` does
  an explicit `if (!userId) return 401` as a defensive belt. Cheap insurance.

### Learnings for future iterations

- **`updatePropertySchema` was missing `type` entirely, not just the new values.** The AC wording "Update schema to accept the new values" was easy to misread as "the old values already work, just widen the enum". In fact no PUT caller has ever been able to change a property's type — LL-019 only updated the create path + search path, because `z.nativeEnum(PropertyType)` auto-propagates to any `createPropertySchema` + `propertySearchSchema` usage, but `updatePropertySchema` didn't have a `type` field at all. When widening an enum, always verify EVERY schema that should accept the new values individually — `grep -n "nativeEnum(PropertyType)" src/utils/` tells you which ones already flow automatically and which ones need a hand-added field.

- **Adding new demo properties is strictly safer than mutating existing ones.** Changing `DEMO_PROPERTY_RJ_4_ID`'s type from `APARTMENT` to `PENTHOUSE` (the property IS semantically a cobertura) would be semantically correct, but downstream tests/seeds may grep or inspect demo property types by id — the mutation would be a silent behavioral change on every existing fixture consumer. Adding 4 new ids with new types (`DEMO_PROPERTY_KITNET_ID` etc.) is purely additive: existing tests continue to see the same 5 demos they always did, plus the 4 new ones are available to any future test that wants them.

- **Keep id registries in sync across 3 files.** When adding a new demo property id: (1) declare it in `prisma/demoIds.ts`; (2) consume it in `prisma/demoData.ts` for the actual property record; (3) add it to `tests/seed.test.ts DEMO_PROPERTY_IDS` AND `tests/demoIds.test.ts expectedConstantNames` so the "ids are valid UUID-v4 and demo-or-random" guards cover them. Forgetting step 3 gives you a green seed.test.ts (because `idIsDemo || idMatchesV4` falls through to the regex branch when the id isn't in the set) but a red demoIds.test.ts (which iterates every exported constant and expects it to be in the allowlist). Two different failure modes, one per test — the pair is the regression tripwire.

- **`z.nativeEnum(PrismaEnum)` is the "automatically accept every value" escape hatch.** When a future rollout widens a Prisma enum (e.g., a new `VisitStatus.RESCHEDULED`), Zod schemas using `z.nativeEnum(VisitStatus)` will auto-accept the new value on the next `prisma generate` — no schema edit needed. Schemas using hand-rolled `z.enum(['SCHEDULED', 'CANCELLED', 'COMPLETED', 'NO_SHOW'])` allowlists will NOT — they'll reject the new value with `VALIDATION_ERROR`. Audit with `grep -rn "z.enum(\[" src/utils/` when proposing any enum widening. Project convention is `z.nativeEnum`; any hand-rolled allowlist is technical debt that should be migrated to nativeEnum opportunistically.

- **`ALTER TYPE ADD VALUE` cannot run inside a transaction block.** The AC called this out explicitly, and LL-019's migration respects it. If you ever see `BEGIN;` / `COMMIT;` wrapping an ALTER TYPE ADD VALUE, Postgres will reject the migration with `ALTER TYPE ... ADD cannot run inside a transaction block`. Prisma's migrate runner runs each migration.sql inside a single implicit transaction by default, so hand-authored enum widening migrations must contain ONLY the ALTER TYPE statements (one per line) — no other DDL, no BEGIN/COMMIT, no DML. If you need to backfill data after an enum widening, put that UPDATE in a separate follow-up migration file.

## Codebase Pattern updates (US-015)

Appending to the `## Codebase Patterns` section at the top of this file:

- **`z.nativeEnum(PrismaEnum)` is the project-wide convention for enum validation** — used by `createPropertySchema.type`, `propertySearchSchema.type`, `updatePropertySchema.type` (new in US-015), `createContractSchema.documentStatus`, `createVisitSchema.source`, etc. When adding a new Zod schema that touches a Prisma enum, use `z.nativeEnum(TheEnum)` — it auto-tracks any future `ALTER TYPE ADD VALUE` widening via `prisma generate`. A hand-rolled `z.enum([...allowlist])` is a regression tripwire waiting to happen: the next time the enum is widened, that allowlist will silently reject the new value and the AC "new value works end-to-end" test will fail without any obvious clue. Audit periodically with `grep -rn "z.enum(\[" src/utils/`.

- **Enum widening migration constraints**: `ALTER TYPE ADD VALUE` CANNOT run inside a transaction block (Postgres rejects with `ALTER TYPE ... ADD cannot run inside a transaction block`). Prisma's migrate runner wraps each migration.sql in an implicit transaction, so enum-widening migrations must contain ONLY the ALTER TYPE statements — no BEGIN/COMMIT, no other DDL, no DML backfills. If you need to backfill rows against the new value (e.g., UPDATE ... SET status = 'NEW_VALUE' WHERE ...), that must be a SEPARATE follow-up migration file after Postgres has committed the enum-value addition. Pattern lives in `prisma/migrations/20260508040000_add_property_type_extended_values/migration.sql`.

- **Add new demo ids rather than mutating existing ones.** Changing an existing demo property's type/status/etc. is a silent behavioral change for every test that grep-inspects the fixture by id. The safe alternative is to add new ids (`DEMO_PROPERTY_KITNET_ID` etc.) — purely additive, existing tests continue to see the same fixtures they always did. When doing this: (1) declare in `prisma/demoIds.ts`, (2) consume in `prisma/demoData.ts`, (3) add to `tests/seed.test.ts DEMO_PROPERTY_IDS` AND `tests/demoIds.test.ts expectedConstantNames` (both — they test different guards).

- **Verify Zod schema coverage when widening an enum.** `z.nativeEnum(PropertyType)` auto-propagates the new values to every schema that uses the wrapper — but only if the schema actually has a field for that enum. `updatePropertySchema` did not have a `type` field at all before US-015, so widening the PropertyType enum in LL-019 didn't help the PUT path. When adding a new value to a Prisma enum, `grep -l "nativeEnum(TheEnum)" src/utils/` tells you which schemas auto-pick it up, but the real check is "which endpoints should accept this field?" — audit the routes, not just the schemas.

### Learnings for future iterations

- **Check if POST validation is newer than PUT validation.** This pattern showed up in US-015 (`updatePropertySchema` didn't have `type` at all when the enum widened) and US-016 (the validation + search wiring was already complete — only GET + seed were gaps). Before writing code for a "POST/PUT/GET accept X" AC, `grep -n "X" src/utils/propertyValidation.ts src/utils/searchValidation.ts src/services/propertyService.ts` and list exactly which surfaces still need work. Often 5 of 7 AC items are pre-built and only 1-2 are genuinely missing.
- **Mock fire-and-forget side-effects when testing GET routes.** `propertyController.getById` at propertyController.ts:120-128 calls `profileViewService.record` (if `?inspectLandlord=true`) and `propertyViewService.record` unconditionally — both are `void`-awaited ("fire and forget"). A plain mock of `getPropertyById` alone will make the DB-hitting services throw when the test mock doesn't stub them. Always add `vi.mock('../src/services/propertyViewService', () => ({ propertyViewService: { record: vi.fn() } }))` + the same for profileViewService to keep the GET test hermetic.

- **`multipartBoolean` + `stringToBoolean` are two different coercion primitives** in this repo. `multipartBoolean` (in `src/utils/propertyValidation.ts`) is used in POST/PUT bodies where multipart form-data sends everything as strings. `stringToBoolean` (in `src/utils/searchValidation.ts`) is used in query string parsing. They both accept `'true'`/`'false'` and reject garbage like `'yep'` with VALIDATION_ERROR. When adding a new boolean field to a Property endpoint, use the right primitive for the layer — don't copy from the wrong file.
